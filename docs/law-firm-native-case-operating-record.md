# Law Firm Native Case Operating Record

Status: draft for attorney/domain review

This document proposes the clean native case format for FirmVault-backed law firm workflows in Mission Control. It is based on the first attorney interview pass in `.planning/LAW_FIRM_NATIVE_CASE_INTERVIEW.md` and on the existing FirmVault architectural rules.

It is not final. The attorney/domain owner should correct the legal workflow assumptions before this becomes a data contract.

## Why This Exists

The current FirmVault contains real law firm history that was generated by humans and imported from prior systems. That history is useful, but it is messy because real case files are messy. The product should not be designed around the accidental shape of a migrated case.

The product should be designed around the shape of a clean case that starts inside this system from day one.

The goal is a native case operating record:

- predictable enough for agents to work safely
- human-readable enough for lawyers and paralegals to inspect
- Git-tracked enough for audit, review, rollback, and accountability
- Obsidian-compatible enough for wikilinks and case navigation
- strict enough that workflows know where to read and write
- flexible enough for real personal-injury practice

## Core Principles

### FirmVault Is The Operating Record

FirmVault is not just a document dump. It is the working, Git-tracked operating record for the case.

The vault stores:

- structured case facts
- markdown shadows of documents
- workflow outputs
- wikilinks between case entities
- activity logs
- audit evidence
- generated draft/shadow documents

The vault does not replace the firm's real document storage.

### The Vault Is A Shadow Of Real Files

Real signed PDFs, email attachments, faxes, audio, scans, Word files, and other original documents live in the firm's own storage system.

FirmVault stores a masked markdown projection of those documents plus links back to the original real-file location.

For example, a received medical bill may have:

- a real PDF in the attorney file system
- a markdown shadow in FirmVault
- frontmatter linking to the original PDF location
- wikilinks to the case, provider, payer, lien, and workflow that processed it

### Git Is The Deliberate Interface

Every agent action should be reviewable as a diff.

For version 1, every task, recipe, and workflow should end in a PR-style review before it becomes final. The runner works in a case worktree, proposes changes, and presents:

- what it did
- what files changed
- what documents it generated
- what facts it proposes to update
- what evidence supports those updates
- what uncertainties or questions remain

Nothing should silently update the canonical case record.

### Logs Are Append-Only

Logs are immutable. Most other working records can be edited, but the edit itself must be logged.

Append-only:

- activity logs
- workflow logs
- audit logs
- task/session logs

Editable with audit:

- provider status fields
- medical chronologies
- lien amounts
- settlement numbers
- client/contact facts
- generated markdown shadows
- case working documents

Git history, activity logs, workflow logs, and PR-style diffs together should make it possible to reconstruct what happened.

### The Attorney Defines The Domain, The System Defines The Contract

Attorney input is authoritative for how personal-injury cases actually work. Technical layout choices still need engineering judgment.

The schema should not blindly copy a prior practice-management-system tab layout. It should preserve the legal workflow and human mental model while choosing file and runtime structures that are durable for agents.

## High-Level Architecture

```text
Real document storage
  PDFs, scans, signed contracts, emails, faxes, Word files
        |
        | document pipeline
        v
FirmVault Git repo
  masked markdown shadows, structured facts, wikilinks, activity logs
        |
        | workflow engine reads/writes through worktrees
        v
Mission Control SQLite runtime
  workflow instances, node state, timers, task sessions, runner audit
        |
        | PR-style review/promotion
        v
FirmVault canonical branch
```

The durable boundary:

- FirmVault holds the legal operating record.
- Mission Control SQLite holds runtime state and task/workflow execution history.
- Real document storage holds originals and non-markdown legal files.

## Case Root Files

### `<case-slug>.md`

The root case file is the human-readable case index and high-level summary.

It should include:

- case identity
- client display name
- case type
- case status/phase
- date of incident
- jurisdiction
- key landmarks
- links to the major ledgers
- concise rollups of insurance, providers, liens, and litigation status

It should not become a dumping ground for every detail.

### `state.yaml`

`state.yaml` should hold compact machine-readable case state that workflows need frequently.

Potential contents:

```yaml
schema_version: 1
case_slug: abby-sitgraves
case_type: auto_accident
status: treatment
landmarks:
  intake_complete:
    satisfied: true
    satisfied_at: "2026-04-25T09:12:00Z"
    evidence:
      - client/intake.md
      - client/contracts.md
  medical_auth_signed:
    satisfied: true
    evidence:
      - client/authorizations.md
```

Open design issue: this should not duplicate every fact in every ledger. It should hold the facts needed for broad phase/workflow routing, while detailed facts live in their ledger files.

## Client Ledger

The client ledger holds intake, contracts, and authorizations.

```text
client/
  intake.md
  contracts.md
  authorizations.md
```

The ideal clean case starts after the intake packet is complete, but the system must allow exceptions.

Minimum case-opening facts:

- name
- address
- phone number
- email
- date of birth
- social security number reference
- case type
- incident facts
- work/employment information
- missed work information
- injuries
- whether the client has seen a doctor or hospital
- known medical providers

PII rule: raw Social Security numbers and other high-risk identifiers should not be stored in the vault. The vault should store masked values or secret references that deterministic rendering code can resolve outside the model context.

## Insurance Ledger

Insurance should be its own ledger because personal-injury cases often have multiple carriers and coverage roles.

```text
insurance/
  pip-<carrier-slug>.md
  bi-<carrier-slug>.md
  um-<carrier-slug>.md
  uim-<carrier-slug>.md
```

Each insurance file should track:

- coverage type: PIP, BI, UM, UIM
- carrier
- claim number
- adjuster
- contact details or links
- status
- limits if known
- coverage-specific workflow facts
- correspondence links
- activity links

This design should support multiple carriers and multiple claims of the same coverage type when real cases require it.

## Contact Cards

The master contact-card model should remain.

There are two levels:

- master contact card, outside the case
- case-specific contact stub, inside the case

Example:

```text
Contacts/Medical/university-of-louisville.md
cases/<case-slug>/contacts/university-of-louisville.md
```

The case stub links to the master contact card. The master card should also expose the cases where that entity appears.

The case-specific record should track the relationship to this case. The master card should track reusable information about the entity.

## Medical Provider Ledger

The medical provider ledger is central to personal-injury work.

The clean provider record should not duplicate general provider identity data if that already exists in the master contact card. It should link to the master provider and track provider-in-this-case facts.

Proposed provider folder:

```text
medical-providers/<provider-slug>/
  provider.md
  treatment.md
  records-bills.md
  chronology.md
  notes.md
```

### `provider.md`

Purpose: case-specific provider index.

Tracks:

- provider display name
- provider slug
- link to master contact card
- case relationship
- provider-specific request rules
- links to treatment, records/bills, chronology, documents, notes, and activity

Example fields:

```yaml
schema_version: 1
provider_slug: uofl-physicians-orthopedics
provider_name: UofL Physicians - Orthopedics
master_contact: Contacts/Medical/uofl-physicians-orthopedics
request_rules:
  records_and_bills_separate: true
  preferred_records_method: fax
  preferred_bills_method: fax
```

### `treatment.md`

Purpose: treatment status and date range.

Tracks:

- treatment start date
- treatment end date
- treatment status
- client-reported treatment updates
- provider-specific notes during treatment

When treatment ends for a provider, the provider-specific records/bills request workflow becomes eligible.

### `records-bills.md`

Purpose: request, follow-up, receipt, payment, and balance facts.

Tracks:

- records request generated
- bills request generated
- authorization/HIPAA used
- records request sent date
- bills request sent date
- send method
- confirmation or evidence
- next follow-up date
- records received date
- bills received date
- records completeness review
- total billed
- paid by Medicare
- paid by Medicaid
- paid by private health insurance
- paid by client
- paid by PIP
- total paid
- remaining balance
- payer information that may reveal liens

The remaining balance should be computed, not manually trusted:

```text
remaining = total_billed - medicare_paid - medicaid_paid - private_health_paid - client_paid - pip_paid - other_paid
```

### `chronology.md`

Purpose: provider-specific chronology.

Tracks each visit/date of service:

- date
- provider/location
- complaints
- diagnoses/assessment
- treatment
- recommendations
- billing reference if relevant
- source record link

This document is editable, but every create/edit event must be logged.

### `notes.md`

Purpose: provider-specific notes that should appear in the provider view.

Notes may be generated by:

- client check-ins
- paralegal review
- records follow-up
- attorney review
- workflow outputs

Important rule: a provider note should also produce or link to an activity log entry so the case-wide history remains complete.

## Records And Bills Workflow Semantics

Provider treatment completion should trigger provider-specific records/bills work.

The workflow should branch based on provider state:

- If records and bills are already received and reviewed, the provider should be complete.
- If requests were already sent but records/bills are missing, the workflow should enter follow-up/wait behavior.
- If requests have not been sent, the workflow should start with authorization/request preparation.
- If the provider requires separate records and bills requests, the workflow should generate separate requests.
- If the provider accepts one combined request, the workflow may generate one combined request.

The first follow-up is an early safety check around 15 days. It is meant to catch failed requests, bad addresses/faxes, returned letters, provider acknowledgements, or provider-specific delay notices. Later follow-ups can follow the configured schedule, such as 30, 45, and 60 days.

## Lien Ledger

Liens should remain their own ledger.

```text
liens/<lien-holder-slug>.md
```

Lien records should track:

- lien holder
- lien type
- source of identification
- related provider or payer
- opening letter sent
- acknowledgement
- amount claimed
- final amount requested
- final amount received
- negotiation status
- resolved amount
- payment status
- evidence links

Medical bills can reveal liens. If bill review shows a payer such as Medicare, Medicaid, private health insurance, or another plan, the system should check known liens. If the payer is new, it should create or propose a lien record and trigger lien workflows.

## Litigation Ledger

Litigation is large enough to be its own area and workflow family.

```text
litigation/
  litigation.md
  discovery.md
  interrogatories.md
  depositions.md
  motions.md
  trial.md
  research.md
```

This ledger should track:

- filing status
- court
- judge
- parties
- service
- deadlines
- discovery
- interrogatories
- depositions
- motions
- trial schedule
- litigation research

Litigation workflows should not be forced into the simpler pre-litigation treatment/demand workflow model.

## Activity Log

Activity is append-only.

Every meaningful case action should create an activity entry or link to one.

Examples:

- client call
- provider note
- records request generated
- records request sent or prepared for human sending
- records received
- bills received
- chronology created
- chronology edited
- lien discovered
- demand drafted
- settlement authority discussed

If work happens inside a ledger-specific view, such as a provider note, the case-wide activity log should still reflect it.

## Documents

Cases are document-heavy. The native design should assume a large number of documents.

Proposed document categories:

```text
documents/
  incoming/
  generated/
  sent/
  shadows/
```

Document records should include frontmatter such as:

```yaml
schema_version: 1
document_id: <stable-id>
case_slug: <case-slug>
document_type: medical_bill
source: mailroom
original_file_uri: <firm-storage-uri>
shadow_of: <original-file-id-or-uri>
related_provider: uofl-physicians-orthopedics
related_workflow_instance_id: 123
received_at: "2026-04-25T08:15:00Z"
masked: true
```

Physical document location should be chosen for reliability, not nostalgia. The UI/provider views can present provider-specific documents through wikilinks and metadata even if documents physically live in a common documents folder.

## Workflow And Task Runtime

Tasks and workflows should be visible from the case, but they do not have to be primary vault files.

Recommended split:

- Workflow definitions live as YAML in Mission Control/FirmVault workflow configuration.
- Workflow instances, nodes, timers, task IDs, sessions, and runner attempts live in Mission Control SQLite.
- Accepted workflow outputs write to FirmVault through PR-style review.
- Activity summaries and audit references write to the vault where appropriate.

The workflow runtime is operational state. The vault is the legal/case operating record.

## Recipe Output Contract

Every recipe should declare:

- allowed reads
- allowed writes
- expected output files
- facts it may satisfy
- logs it must write
- evidence it must attach
- human review requirement
- external action policy

For version 1, no final workflow action should bypass review.

Example for a medical records request recipe:

```yaml
recipe: firmvault-medical-records-prepare-request
subject: law_firm_case_provider
reads:
  - cases/<case>/client/authorizations.md
  - cases/<case>/medical-providers/<provider>/provider.md
  - cases/<case>/medical-providers/<provider>/treatment.md
writes:
  - cases/<case>/documents/generated/<request-shadow>.md
  - cases/<case>/medical-providers/<provider>/records-bills.md
  - cases/<case>/activity/<activity-entry>.md
external_actions:
  version_1: none
review:
  required: true
completion_evidence:
  - generated request shadow
  - linked authorization
  - proposed follow-up date
```

## Human Review And External Sending

For version 1, humans are in the loop everywhere.

Medical records request example:

1. Agent prepares records/bills request documents.
2. Agent creates markdown shadows and proposed provider updates.
3. Agent records proposed follow-up dates.
4. Agent submits work for PR-style review.
5. Human approves or rejects.
6. Human sends the actual request in this iteration.

Future versions may allow agents to send through APIs such as Lob, DocuSign, fax, or email providers. Even then, the system should separate:

- preparing a document
- approving a document
- sending a document
- recording proof of sending

## Incoming Document Pipeline

The mailroom/import pipeline should be separate from the workflow interpretation layer.

```text
mailroom folder
  -> watcher
  -> bulk PDF/document intake
  -> markdown conversion
  -> light PII masking
  -> split into original documents
  -> case matching
  -> naming
  -> FirmVault shadow filing
  -> original PDF filing in real document storage
  -> markdown frontmatter points to original file
  -> Git diff remains uncommitted
  -> review/interpreter agent processes uncommitted changes
```

Known pieces:

- Kreuzberg can convert documents to markdown.
- A future masking layer should remove or reference high-risk PII.
- A QMD-style enrichment layer can add local embeddings to markdown.
- A daily or periodic review agent can inspect uncommitted document changes and update case facts.

The import pipeline should not by itself decide final legal meaning. It creates shadows and candidates. A later review/interpretation workflow updates provider, lien, insurance, or other case facts.

## Fact Resolution

Workflows should depend on named facts, not ad hoc agent guessing.

Examples:

```yaml
law_firm.case.intake_complete
law_firm.case.medical_auth_signed
law_firm.provider.treatment_complete(provider_slug)
law_firm.provider.records_requested(provider_slug)
law_firm.provider.bills_requested(provider_slug)
law_firm.provider.records_received(provider_slug)
law_firm.provider.bills_received(provider_slug)
law_firm.provider.chronology_complete(provider_slug)
law_firm.case.all_required_provider_records_complete
law_firm.lien.identified(lien_holder_slug)
```

Each named fact should have a deterministic resolver that knows where to read.

This avoids asking an agent to search the whole vault and infer whether a workflow should start.

## Workflow-Derived Vault Shape

The workflow definitions should drive the clean FirmVault shape. Each workflow should answer two questions:

1. What does the agent need to read without guessing?
2. What should the agent create or update so the next workflow can run without guessing?

The vault layout is therefore not just a filing preference. It is the interface between legal work, agents, humans, and audit.

### Accident Report Workflow

The existing accident-report workflow confirms the first concrete shape for a clean auto-accident file.

The workflow should be able to read:

- case type and accident type
- whether police were called
- accident date, time, and location if known
- intake document shadow
- signed contract status
- signed HIPAA/authorization status
- known parties from intake
- known insurance facts from intake
- existing accident report status, if the report was already obtained

The workflow should create or update:

- accident report request status
- reporting agency
- report number if known
- report requested date
- report received date
- accident report markdown shadow
- original-file link for the report PDF
- extracted client unit/role
- extracted at-fault party facts
- extracted BI insurance facts
- extracted PIP insurance/waterfall facts
- witness contacts
- officer/citation/contributing-factor facts
- liability assessment and red flags
- comparison between client intake story and police-report facts
- activity log entries
- workflow/task audit summary

It should trigger or unblock:

- PIP claim setup
- BI claim setup
- UM/UIM analysis if BI is absent, unclear, or insufficient
- contact-card creation or update for parties/witnesses
- attorney review if liability is disputed, the client is cited, insurance is missing, or the report conflicts with intake

The old workflow references FalkorDB, Cypher, graph state, and per-case JSON files. Those are implementation fossils. In the native system, the outputs should write to the case vault contract and Mission Control workflow state.

```text
    <at-fault-party-slug>.md
    <witness-slug>.md

  insurance/
    pip-<carrier-slug>.md
    bi-<carrier-slug>.md
    um-<carrier-slug>.md
    uim-<carrier-slug>.md

  documents/
    incoming/
      <accident-report-shadow>.md

  activity/
    <YYYY-MM-DD-HHMM>-accident-report.md
```

Design implication: an `accident/` ledger may be warranted. Without it, accident facts would be scattered across intake, insurance, contacts, and activity logs. The workflow needs one predictable place to summarize the operative accident facts and one predictable place to store police-report-specific extraction.

### PIP Claim Setup Workflow

The existing PIP workflow is a good example of why insurance cannot be just a carrier contact card. The workflow has to track coverage determination, claim setup, document generation, follow-up, payment readiness, and later PIP monitoring.

The workflow should be able to read:

- case type and jurisdiction
- accident date/time/location
- police report extraction, if available
- intake facts about the vehicle occupied by the client
- whether the client was driver, passenger, pedestrian, or other
- whether the client was on the title of the occupied vehicle
- whether the occupied vehicle was insured
- occupied vehicle insurer, policy number, and insured/owner if known
- whether the client has their own auto insurance
- client insurer and policy number if known
- whether a resident household member has auto insurance
- household member insurer, policy number, and relationship if known
- signed contract status
- signed HIPAA/authorization status
- client identity/contact facts needed for the KACP application
- injury description and initial provider facts from intake
- existing PIP claim record, if one already exists

The workflow should create or update:

- PIP waterfall input facts
- PIP waterfall determination
- waterfall path/reasoning
- PIP carrier type:
  - occupied vehicle insurer
  - client's own insurer
  - household member insurer
  - KAC
  - disqualified
- disqualification reason, if applicable
- PIP carrier/contact link
- PIP policy number
- PIP claim number
- KACP application shadow/document
- PIP letter of representation shadow/document
- application sent/submitted date
- LOR sent date
- claim opened date
- claim acknowledged flag/date
- adjuster name, phone, email, and mailing/fax instructions
- coverage limit
- deductible if any
- billing instructions
- ready-to-pay-bills flag/date
- follow-up timers
- activity log entries
- workflow/task audit summary

It should trigger or unblock:

- medical-provider billing instructions
- bill submission/payment monitoring
- PIP exhaustion monitoring
- health-insurance/lien workflows if no PIP or PIP is disqualified
- BI-only focus if PIP is disqualified
- KAC follow-up if assigned claims is required
- attorney review if PIP is unclear, disputed, denied, or disqualified

### BI Claim Setup Workflow

The BI workflow shares the same basic claim-ledger pattern as PIP, but its key branch is different. PIP turns on waterfall and payment readiness. BI turns on at-fault party identification, liability status, policy limits, and possible additional claims.

The workflow should be able to read:

- case phase/status
- signed contract status
- signed HIPAA/authorization status
- accident report extraction, if available
- accident facts from intake if no report is available yet
- at-fault party identity
- at-fault party vehicle/unit information
- at-fault party insurance company
- at-fault party policy number if known
- known property-damage claim information
- existing BI claim records
- existing UM/UIM records, if any
- liability facts from police report, citations, contributing factors, and narrative

The workflow should create or update:

- BI carrier/contact link
- insured name
- at-fault party/contact link
- policy number
- claim number
- claim opened date
- BI letter of representation shadow/document
- LOR sent date
- claim acknowledged flag/date
- adjuster name, phone, email, and mailing/fax instructions
- liability status
- liability percentage
- liability denial/partial-fault reason
- date liability determined
- policy limits disclosed flag/date
- coverage limit per person
- coverage limit per accident
- UM/UIM limits disclosed by BI carrier if included in response
- umbrella/excess policy indicators
- follow-up timers
- activity log entries
- workflow/task audit summary

It should trigger or unblock:

- UM/UIM analysis if liability is denied, partial, unclear, or limits appear insufficient
- additional BI claim workflows if multiple liable parties or vehicles exist
- attorney review if liability is disputed or comparative fault is asserted
- demand preparation later once treatment/records/bills are complete
- coverage follow-up if policy limits are not disclosed within the expected period

### UM And UIM Coverage Tracks

UM and UIM should be represented in the insurance ledger even though they are conditional tracks. They are not merely litigation theories. They can require early notice, preservation steps, claim setup, and later complaint/demand handling.

UM becomes relevant when:

- the at-fault driver has no insurance
- the at-fault driver is unknown
- the at-fault driver fled the scene
- liability coverage is denied, lapsed, unavailable, or insolvent
- the client has UM coverage through their own policy

UIM becomes relevant when:

- the at-fault driver has BI coverage
- the BI limits may be insufficient to compensate the client
- the client has UIM coverage through their own policy
- BI settlement is being considered
- UIM rights must be preserved before accepting BI settlement

The existing workflow manual specifically calls out UIM preservation through a COOTS letter. That means UIM cannot be treated as an afterthought at settlement. The workflow system should make the preservation step visible before BI settlement is approved.

The UM/UIM workflow should be able to read:

- BI liability status
- BI limits
- damages estimate/demand value when available
- accident report extraction
- whether the at-fault driver is uninsured, unknown, or underinsured
- client policy information
- client UM/UIM coverage limits
- policy notice requirements
- whether timely notice has already been sent
- whether BI settlement is pending
- whether a COOTS/preservation letter is required or sent

The UM/UIM workflow should create or update:

- UM/UIM carrier/contact link

- conditions-precedent checklist

- relationship to underlying BI claim
- attorney-review flags
- activity log entries
- workflow/task audit summary

### Negotiation, Offer Evaluation, And Claim Response Workflows

Source workflows reviewed:

- `phase_4_negotiation/workflows/track_offers`
- `phase_4_negotiation/workflows/offer_evaluation`
- `phase_4_negotiation/workflows/negotiate_claim`
- `offer-tracking`
- `offer-evaluation`
- `negotiation-strategy`

These workflows start after a demand is sent. They do not behave like a simple checklist. Negotiation is an event loop around external responses from the carrier, defense attorney, client, and attorney.

The source material divides negotiation into three concerns:

1. **Track offers.** Keep a complete history of every demand, offer, counter, rejection, acceptance, deadline, and status.
2. **Evaluate offers.** Calculate the actual client recovery and compare the offer to demand, specials, limits, liens, liability, litigation risk, and case value.
3. **Negotiate the claim.** Decide whether to accept, counter, reject, answer an information request, escalate, or declare impasse.

For the native vault, this means negotiation should not be a single case-level status. It should be claim-specific because one case can have multiple active negotiations:

- BI claim against one carrier.
- BI claim against another at-fault party or carrier.
- UM claim against the client's carrier.
- UIM claim after BI limits are insufficient.
- Less common claims such as Med Pay or workers' compensation, when relevant.

The negotiation workflow should not have to rediscover demand facts from a PDF or activity log. It should read the claim ledger and the demand readiness/send records.

#### Response Types

The workflow waits for one of these response types:

| Response | Meaning | Workflow consequence |
| --- | --- | --- |
| Acknowledgment | Carrier confirms demand received and under review | Log acknowledgment, update receipt status, keep 30-day follow-up |
| Information request | Carrier asks for records, bills, authorizations, clarification, or other material | Evaluate request, provide/prepare response, log, schedule follow-up |
| Offer | Carrier makes settlement offer | Track offer, evaluate offer, attorney review, client authority, response |
| Denial | Carrier denies liability/coverage or refuses payment | Attorney review, litigation/coverage/UM/UIM/closing branch |
| No response | Deadline or follow-up date passes with no substantive response | Follow up, escalate to supervisor, document attempts |

This maps cleanly onto workflow nodes:

```yaml
nodes:
  - wait_for_response
  - classify_response
  - track_offer
  - evaluate_offer
  - attorney_review
  - client_authority
  - send_counter_or_acceptance
  - follow_up
  - settlement_reached
  - impasse_review
```

#### Attorney Review And Client Authority

Attorney recommendation and client authority are separate records.

- Was the offer explained to the client?
- Was net-to-client explained?
- Was the attorney recommendation communicated?

#### Settlement Or Impasse

Negotiation exits in one of two main ways.

If settlement is reached:

```yaml
settlement:
  reached: true
  date:
  amount:
  claim:
  accepted_offer_id:
  client_authorized:
  terms:
  next_workflow: settlement_processing
```

This does not mean the case is done. It triggers settlement processing: releases, final liens, settlement statement, trust/accounting, disbursement, and closing.

If negotiation fails:

```yaml
impasse:
  reached: true
  date:
  reason:
  last_offer:
  last_counter:
  liability_denial:
  coverage_denial:
  attorney_review:
  client_decision:
  next_path: litigation | more_treatment | close_decline | further_review
```

Impasse should not simply mark the workflow complete. It should create the next review branch:

- File suit / complaint workflow.
- Return to treatment if the case value is underdeveloped and the client still needs care.
- Coverage review / UM or UIM branch if the problem is available insurance.
- Closing/decline if litigation is not viable or client does not want to proceed.

#### Suggested Native Layout

Negotiation can be represented with a combination of claim-level ledgers and optional case-level summaries.

Claim ledgers remain primary:

```text

  um-state-farm.md
  uim-state-farm.md
```

Each claim ledger should contain or link:

- demand sent data
- negotiation status
- offer history
- current offer
- current counter
- current follow-up
- settlement/impasse result

A case-level negotiation folder can make attorney review easier:

```text
negotiation/
```

Design choice: the claim ledger should be the machine-readable workflow source; the `negotiation/` folder can be the human-readable work product and durable analysis library.

#### Workflow Implications

The materializer should not create every possible negotiation task up front. It should expose tasks based on events:

- Demand sent creates follow-up timers.
- Receipt acknowledgment satisfies the receipt-check task.
- Offer received creates offer-tracking and offer-evaluation tasks.
- Completed offer analysis creates attorney review.
- Attorney recommendation creates client-authority task.
- Client authority creates response/counter/acceptance task.
- Counter sent creates the next wait/follow-up timer.
- Acceptance creates settlement-processing workflow.
- Denial/impasse creates litigation or attorney-review branch.

The workflow therefore needs both time-based nodes and event-based nodes:

```yaml

  - demand_sent
  - demand_receipt_confirmed
  - no_response_after_30_days
  - offer_received
  - info_request_received
  - denial_received
  - client_authority_recorded
  - counter_sent
  - settlement_accepted
  - impasse_declared
```

This is another place where the push-style dependency model matters. When an offer is recorded, any workflows depending on `offer_received` should become eligible immediately without a broad scan over every case.

### Settlement Processing And Lien Negotiation Workflows

Source workflows reviewed:

- `phase_5_settlement/workflows/settlement_processing`
- `phase_5_settlement/workflows/lien_negotiation`
- `settlement-statement`
- `docusign-send`

Settlement begins after a specific offer has been accepted. It should not be represented as only a case-level phase flip. It should point back to the accepted offer, the claim being settled, the settlement amount, and the authority records that allowed acceptance.

Settlement differs from negotiation because money and releases are now involved. The workflow has more hard gates:

- Client authorization to settle.
- Attorney approval of release language.
- Client signature on release.
- Settlement check received.
- Funds deposited into trust.
- Funds cleared.
- Lien payments or approved trust holdbacks.
- Client distribution.

The settlement phase can close only when the client has received their distribution and all liens are either resolved or moved into a proper lien/final-distribution holdback path.

The initial settlement statement is the first gross-to-net accounting after settlement is accepted.

It should read:

- accepted settlement amount
- signed fee agreement
- fee percentage and fee calculation
- case expenses
- lien inventory
- final lien amounts where known
- estimated or disputed lien holdbacks where final amounts are not known

```text
gross settlement
- attorney fee
- case costs
- lien payments or lien holdback

```

Important distinction:

- The initial settlement statement can include lien holdbacks.

#### Authorization To Settle

The authorization to settle is the client's explicit approval of the settlement and distribution.

It should record:

- gross settlement amount
- deductions
- expected net
- lien holdback
- statement that holdback may later produce a supplemental distribution
- client signature
- date signed
- delivery method

This is a separate gate from oral authority during negotiation. Oral authority may accept the offer; signed authorization approves the settlement accounting and distribution.

#### Release Processing

Release handling is not just "client signs release." The release needs review.

The release review should check:

- correct client name
- correct defendant/released party names
- correct carrier/claim number
- correct settlement amount
- correct accident/date/matter
- scope of release
- confidentiality terms
- indemnity/hold-harmless language
- Medicare/Medicaid language
- dismissal language if litigation is pending
- whether UM/UIM, Med Pay, PIP, or other claims are accidentally released
- whether liens/subrogation obligations are misstated

Suggested release review:

```yaml
  reviewer:
  approved:

  requested_changes:
  final_release_path:
  client_signature_required:
  signed_date:
  returned_date:
```

If release language is problematic, the workflow should branch to attorney review and carrier negotiation before the client signs.

1. Provider LOPs.
2. Hospital liens.
3. Fully insured health plans.
4. Self-funded ERISA plans.
5. Medicare.
6. Medicaid.

Reduction arguments vary by lien type:

- Common fund doctrine.
- Made whole doctrine.

- Disputed liability.
- Limited recovery / policy limits.
- Immediate payment.

#### Distribution

Distribution should happen only after settlement funds clear.

Distribution categories:

- attorney fee
- case cost reimbursement
- lien payments
- client net distribution
- disputed lien holdback

The client-paid landmark is a hard blocker. The case cannot close until the client distribution is issued and receipt/delivery is recorded under the applicable office rule.

Conceptual distribution record:

```yaml
distribution:
  settlement_statement_path:
```

#### Supplemental Statement / Lien Phase Branch

If all liens are resolved before client distribution, the settlement phase can close after client payment.

If one or more liens remain disputed or unresolved:

1. Undisputed liens are paid.
2. Approved disputed amounts are held in trust.
3. Client may receive a partial distribution if approved.
4. The case moves into lien/final-distribution workflow.
5. Once liens resolve, a supplemental settlement statement reconciles the original holdback against actual payments.
6. Any remaining funds are distributed to the client.
7. Trust balance is zeroed.

This means `client_received_funds` and `all_liens_resolved` are related but not identical. A client may receive an initial distribution while a lien holdback remains.

#### Workflow Implications

The materializer should create settlement tasks sequentially:

- accepted offer creates settlement statement / authorization task.
- signed authorization creates confirm-settlement-and-request-release task.
- release received creates release review task.
- approved release creates client signature task.
- signed release creates return-release task.
- returned release creates settlement-check wait/follow-up task.
- check received creates deposit/clearance task.
- cleared funds create lien payment and distribution tasks.
- unresolved liens create lien-phase tasks.
- client distribution and lien resolution create closing eligibility.

This is a strong example for human gates and external waiting nodes. Many steps are not "agent does all work"; they are agent prepares, tracks, verifies, and prompts the human at the right gate.

### Lien Phase And Final Distribution Workflows

Source workflows reviewed:

- `phase_6_lien/workflows/get_final_lien`
- `phase_6_lien/workflows/negotiate_lien`
- `phase_6_lien/workflows/final_distribution`
- `supplemental-statement`

The lien phase is the post-settlement branch for unresolved liens and trust holdbacks. It should not replace the general lien lifecycle. Earlier workflows identify liens, request preliminary amounts, track lien holders, and estimate exposure. Phase 6 is narrower: settlement has occurred, money or holdback exists, and final closing is blocked by unresolved lien accounting.

The phase has two hard blockers:

1. All liens resolved and paid.
2. Final distribution completed.

The case enters this branch when settlement processing cannot fully close because liens remain unresolved.

Entry examples:

- Medicare final demand has not arrived.

- ERISA plan asserts full reimbursement.

- Provider lien needs negotiation.
- Hospital lien amount is disputed.
- Multiple liens exceed available holdback.
- Client received an initial distribution, but part of the settlement remains in trust.

#### Get Final Lien Amount

This workflow runs per unresolved lien.

It should:

1. Confirm the lien holder and lien type.
2. Confirm settlement amount/date and client identifiers needed for the lien holder.
3. Generate or prepare the final lien request.
4. Submit through the correct channel or prepare for human submission.
5. Record request date.
6. Set follow-up timer.
7. Record final amount when received.
8. Record payment deadline.
9. Store/link final demand or final statement document.

Lien type differences matter:

- Medicare: final demand after settlement information, federal priority, payment deadline.
- Medicaid: state-specific final amount and reduction procedure.
- ERISA: final statement plus plan documents if reduction is disputed.
- Hospital/provider: final balance and lien filing status.
- Workers' compensation: subrogation and any special settlement considerations.

#### Negotiate Lien

This workflow becomes eligible after a final amount is known and negotiation is warranted.

Negotiation should evaluate:

- net to client if lien paid in full
- settlement amount
- attorney fees and costs
- other liens
- policy limits or limited recovery
- liability disputes
- hardship
- lien holder flexibility
- legal basis for reduction

Reduction arguments may include:

- procurement-cost reduction
- common fund
- made whole
- statutory formula
- ERISA plan language weakness
- compromise/waiver
- hardship
- limited recovery
- immediate payment

The workflow should produce:

- reduction request
- submitted date
- follow-up date
- response
- negotiated amount
- written confirmation
- final payment amount

It should not mark the lien resolved until the required written confirmation and payment/satisfaction steps are complete.

Payment is its own gate.

For each lien:

```yaml
  satisfaction_requested:
  satisfaction_received:
```

The system should distinguish:

- agreed amount
- paid amount
- satisfaction received

Those are not the same. A lien may be negotiated but not paid, or paid but satisfaction not yet received.

#### All Liens Resolved

`all_liens_resolved` should be derived from lien records.

Conceptual predicate:

```text
all liens are one of:
- paid and satisfaction received
- waived with written waiver
- released with written release
- attorney_resolved with documented authority
```

The system should avoid a separate manual checkbox that can drift from the actual lien files.

#### Final Distribution

Final distribution becomes eligible only after all liens are resolved.

The supplemental statement reconciles:

- original settlement statement
- original lien holdback
- actual lien payments
- waivers/reductions
- remaining trust balance
- additional amount due to client

Core calculation:

```text
original_holdback
- sum(actual_lien_payments)
= surplus_to_client

initial_client_net
+ surplus_to_client
= total_client_net
```

If the actual lien payments exceed the holdback, the workflow needs attorney review before any document is generated. The source skill identifies possible resolutions: attorney covers shortfall from fee, client covers from initial distribution, or another documented choice. That is not an agent-only decision.

#### Workflow Implications

The materializer should create tasks from lien state:

- unresolved lien with no final request creates `get_final_lien`.
- final request sent and no response creates time-based follow-up.
- final amount received creates `negotiate_lien` or `pay_lien`.
- negotiated amount confirmed creates `pay_lien`.
- lien paid but no satisfaction creates satisfaction follow-up.
- all liens resolved creates final distribution.
- final distribution completed creates close-case eligibility.

This phase is a good example of the dependency system needing per-record dependencies, not only case-level landmarks. A case can have five liens where four are complete and one is blocking. The UI should show exactly which lien is blocking final distribution.

### Closed And Archived Workflows

Source workflows reviewed:

- `phase_8_closed/workflows/close_case`
- Phase 8 closed landmarks
- Current reconciler checks for `all_obligations_complete`, closing/final letter, and `archived_date`

Closed is the terminal operating state of a matter. It should not mean only that there are no visible tasks. It should mean the system has verified that all legal, financial, trust, document, client, and archive obligations are complete for the applicable closure path.

The source phase allows several entry paths:

- client received final distribution after settlement
- verdict/judgment resolved and funds distributed
- case declined
- representation withdrawn
- other attorney-approved termination

Those paths should not all use the same checklist. A declined intake does not need lien payment checks. A settled case does.

#### Closure Types

Suggested closure type vocabulary:

```yaml
closure:
  type: settled | litigated_verdict | declined | withdrawn | dismissed | transferred | duplicate | other
  reason:
  initiated_date:
  completed_date:
  attorney_approved:
```

The closure type controls which checklist applies.

#### Successful Settlement Closure

For a settled case, closure should verify:

- settlement funds received
- funds deposited and cleared
- client received initial distribution
- all liens paid, waived, released, or attorney-resolved
- supplemental distribution complete if applicable
- trust balance for the case is zero
- release executed and returned
- settlement statement and supplemental statement stored
- client final letter sent
- physical and digital archive complete

Suggested checklist:

```yaml
obligations:
  settlement_funds_received:
  client_paid:
  all_liens_resolved:
  trust_balance_zero:
  releases_complete:
  court_requirements_complete:
  final_distribution_complete:
  final_letter_sent:
  archive_complete:
```

The `all_obligations_complete` field should be derived from this checklist, not manually set without evidence.

#### Declined Or Withdrawn Closure

Declined and withdrawn matters need a different shape.

They may require:

- decline letter or withdrawal letter
- warning about limitations/deadlines
- statement that representation has ended or never began
- return of original documents
- archive/retention metadata
- attorney approval

Suggested checklist:

```yaml
decline_withdrawal:
  attorney_approved:
  letter_sent:
  limitations_warning_included:
```

The source docs say cases can reopen for a new related matter, newly discovered coverage, appeal, or post-judgment issue, but cannot simply reopen after full settlement/judgment satisfaction as if nothing happened.

- closed cases remain closed
- reopening usually creates a new linked matter

- relevant archived documents can be copied/projected into the new matter

- retention/destruction review
- reopened matter creation
- post-judgment/appeal linkage

The materializer should treat `status: closed` as a hard suppressor for ordinary onboarding, treatment, demand, negotiation, settlement, and lien workflows.

#### Litigation Materializer Behavior

Litigation workflow generation should be event- and deadline-driven.

- Litigation decision creates complaint drafting.
- Complaint filed creates service tasks per defendant.

#### Native-Vault Implication

- Litigation needs a `litigation/` area even for cases that started pre-litigation.
- Litigation deadlines are a source of truth for workflow timers.
- Many litigation records are per-party, per-defendant, per-discovery-set, per-deposition, or per-deadline.
- Settlement from litigation should reuse settlement workflows, but should carry litigation-specific fee rate, dismissal, release, and court-order requirements.
- The workflow system needs to show the active litigation subphase and what is blocking movement to the next subphase.

#### Parallel Defendant Tracks

This is the key structural point in 7.1:

```text
Defendant A: served -> answered -> discovery eligible
Defendant B: served -> answer deadline pending
Defendant C: service failed -> alternate service branch
```

The case can begin discovery against Defendant A while Defendant B or C remains in complaint/service/answer handling.

Therefore, subphase 7.1 has two meanings:

- **Case-level 7.1 open:** at least one defendant still needs service/answer/default handling.
- **Defendant-level 7.1 complete:** that defendant is served and has answered/defaulted.

Discovery materialization should be defendant-aware.

#### 7.1 Workflow Implications

The materializer should create tasks like this:

- litigation decision creates complaint draft task
- complaint attorney-approved creates file complaint task
- filed complaint and summons create per-defendant service tasks
- service attempt failure creates service-escalation task for that defendant
- service completed creates answer-deadline timer for that defendant
- answer received creates answer-analysis task
- answer with counterclaim creates counterclaim-response deadline/task
- no answer by deadline creates default-review/default-motion task
- answer/default complete creates discovery eligibility for that defendant
- all defendants answer/default creates case-level 7.1 complete

This avoids turning multi-defendant litigation into a single blocked pipeline.

#### Written Discovery Completion

Written discovery is complete when:

- our propounded discovery has been served
- responses have been received
- deficiencies are resolved, waived, or attorney-approved for no further action
- discovery served on us has been answered
- production and privilege logs are complete
- RFA risk has been addressed

Suggested derived predicate:

```text
written_discovery_complete =
  all propounded sets complete
  AND all received sets responded
  AND no unresolved high-severity deficiencies
  AND no pending RFA/admission risk
  OR attorney marks substantially complete with reason
```

The "substantially complete" override matters because litigation can move to mediation even when a low-value discovery issue remains open. That should require attorney approval and a reason.

#### Trial Milestones

The 7.5 landmarks are:

- jury selected
- plaintiff's case presented
- all evidence closed
- verdict rendered or other resolution

Suggested trial completion predicate:

```text
trial_concluded =
  verdict rendered
  OR directed verdict entered
  OR mistrial declared
  OR settlement reached during trial
  OR other court resolution entered
```

The verdict/rendered hard blocker should not assume a jury verdict only. The source workflow already allows directed verdict or other resolution.

#### Post-Trial Routing

Trial result determines the next phase:

```text
if plaintiff_verdict:
  route to Phase 5 settlement/collection/distribution
elif settlement_during_trial:
  route to Phase 5 settlement
elif defense_verdict:
  route to closure or appeal evaluation
elif directed_verdict:
  route based on winner and appeal posture
elif mistrial:
  route to retry decision, settlement negotiation, or closure
```

A plaintiff verdict is not the same as ordinary pretrial settlement. It needs collection/judgment handling, possible post-judgment interest, appeal deadlines, costs, and distribution once funds are collected.

A defense verdict should not immediately close without an appeal/post-trial review branch. The attorney should decide whether to appeal, move for new trial, negotiate costs, or close.

- verdict/judgment entered updates verdict-rendered hard blocker
- plaintiff verdict creates collection/settlement workflow
- defense verdict creates appeal/post-trial-review or closure workflow
- mistrial creates retry/settlement/closure decision task
- settlement during trial creates settlement workflow

The trial workflow is mostly event capture and routing. It should preserve enough structured information that the file can later show what happened, what proof came in, what the result was, and why the next workflow was chosen.

## Consolidated Native Case Contract Candidate

| `accident/` | incident facts, police report, liability analysis, defendant facts |
| `insurance/` | PIP, BI, UM, UIM, Med Pay, workers' compensation claim state |
| `medical-providers/` | provider identity, treatment dates, records/bills requests, bills, chronology |
| `liens/` | lien candidates, formal liens, final amounts, negotiation, satisfaction |
| `demand/` | demand readiness, package, letter, exhibits, final sent demand |
| `negotiation/` | offers, counteroffers, client authority, offer analyses |
| `settlement/` | release, settlement funds, trust accounting shadow, distribution |
| `litigation/` | pleadings, service, discovery, mediation, trial prep, trial, verdict |
| `documents/` | markdown shadows and generated/sent document artifacts |
| `activity/` | append-only human-readable activity history |
| `workflow-log/` | append-only workflow summaries and promoted task outcomes |

Each ledger should include enough structured frontmatter or structured markdown tables for workflows to evaluate predicates without inference-heavy text scraping.

### Workflow Predicate Pattern

Workflow predicates should resolve from the case operating record in this order:

1. Explicit case frontmatter landmark, if present.
2. Relevant entity ledger structured state.
3. Linked document shadow evidence.
4. Append-only activity/workflow log evidence.
5. Human review or attorney override, with reason.

Example:

```text
request_final_lien_amount eligible =
  lien identified in liens/<holder>.md
  AND treatment complete for related provider(s)
  AND settlement/demand posture requires final lien amount
  AND no active timer or already-open task exists for the same lien request
```

The materializer should not need to scan thousands of possible hidden tasks. It should look at active workflow instances and dependency events. When a dependency is satisfied, the dependency engine should update the relevant workflow/node state and make the next node eligible.

### Timer Pattern

Timers should be workflow-node dependencies, not separate ad hoc reminders.

Timer fields should store:

```yaml
timer:
  kind: duration | deadline | calendar_date
  reference_event:
  reference_date:
  offset:
  due_at:
  satisfied:
  satisfied_at:
  cancelled:
  cancelled_reason:
```

Examples:

- records request sent + 15 days = first records follow-up
- records request sent + 30 days = second records follow-up
- mediator proposal received + response deadline = proposal-response task
- answer served + counterclaim deadline = response-to-counterclaim task
- expert disclosure deadline minus lead time = expert-disclosure task

If the underlying objective is satisfied early, the timer should be cancelled with a reason rather than producing unnecessary work.

### Workflow Instance Pattern

Each active workflow instance should have:

```yaml
workflow_instance:
  id:
  workflow_id:
  case_id:
  status: active | waiting | complete | cancelled | failed
  trigger:
  started_at:
  completed_at:
  cancelled_reason:
  current_nodes:
  completed_nodes:
  blocked_nodes:
```

The canonical workflow definition can live in Mission Control or workflow YAML. The case vault should hold the promoted summary and legal facts, not every transient runner detail. The full runner session/audit can remain in Mission Control and Git history.

### Recipe Node Pattern

A recipe node should become a task only when all dependencies are satisfied.

Each recipe-backed task should know:

- workflow instance id
- node id
- recipe id
- case id
- workspace/worktree
- allowed tools
- model
- SOUL/system prompt
- review instructions
- expected writes
- expected evidence
- completion predicate

The agent's task output should propose changes through a worktree/PR-style review. Once accepted, the resulting case-file edits satisfy downstream dependencies.

### Human Review Pattern

Human review is a node type, not an exception.

Human review can:

- approve proposed changes
- reject proposed changes
- send task back with comments
- mark a node not applicable
- bypass a workflow branch with reason
- satisfy a landmark with evidence
- close/cancel a workflow instance

Not applicable and bypass decisions should be logged and should satisfy or cancel the relevant dependency intentionally. They should not leave downstream workflows confused about whether work is incomplete.

### Minimum Blank Auto-Accident Case

A clean auto-accident case can start with this minimum shape:

```text
cases/<case-slug>/
  <case-slug>.md
  Dashboard.md
  client/
    intake.md
    contracts.md
    authorizations.md
  accident/
    accident.md
  contacts/
  insurance/
  medical-providers/
  liens/
  documents/
    incoming/
    generated/
    sent/
    shadows/
  activity/
  workflow-log/
```

Initial workflows should usually include:

- intake verification
- accident report acquisition/review
- PIP identification and setup
- BI identification and setup
- UM/UIM/Med Pay/workers' compensation screening as conditional workflows
- medical provider setup for known providers
- client check-in cadence
- treatment monitoring

Conditional workflows should remain hidden until their trigger is satisfied. For example, final lien amount requests should not materialize merely because the lien workflow exists. They should become visible only when the lien exists, treatment is complete, and the case posture requires final numbers.

## Open Design Questions

These should be resolved before this becomes a data contract:

1. Should `client/`, `insurance/`, `medical-providers/`, and `litigation/` be physical folders, or should some be generated views over `documents/`, `contacts/`, and ledger files?
2. Should `state.yaml` remain a compact landmark/phase file, or should broad case facts move into separate ledger files only?
3. How should raw PII references be represented so the vault stays masked while deterministic renderers can fill documents?
4. Should provider records be one file per provider or a small folder per provider?
5. Where should generated Word/PDF artifacts live before human approval?
6. What exact frontmatter should every document shadow carry?
7. How should a workflow instance write a vault-visible audit summary without duplicating Mission Control SQLite runtime state?
8. What is the minimum native case template needed to create a brand-new clean case?
9. Should auto-accident cases have a dedicated `accident/` ledger with `accident.md`, `police-report.md`, and `liability.md`, or should those facts live in the main case file plus document shadows?
10. Should PIP live as one `insurance/pip-<carrier>.md` claim ledger with sections, or should PIP coverage, generated documents, and payment monitoring be split into separate files?
11. Should BI claim files be keyed by carrier only or by carrier plus insured/at-fault party, since one carrier could theoretically represent multiple parties in the same accident?
12. Should UM/UIM be standard optional claim ledgers under `insurance/`, created only when triggered, or should every MVA case have placeholder UM/UIM evaluation fields from day one?
13. Should workers' compensation live under `insurance/`, or should it be a separate `benefits/`/`workers-comp/` ledger because it follows different settlement rules?
14. Should each provider have separate physical files for `provider.md`, `treatment.md`, `records-bills.md`, and `chronology.md`, or one provider file with sections plus generated views?
15. Should the case-level medical chronology be a generated aggregate from provider chronologies, or should it be a primary editable document with provider chronologies as supporting material?
16. Should lien candidates be separate lightweight records before they become formal `liens/<holder>.md` files, or should every credible candidate immediately create a lien file with `status: candidate`?
17. Should final lien amount requests live only in `documents/generated/sent`, or should each lien file also keep a request history table?
18. Should client check-ins live in one append-only `client/check-ins.md`, in separate activity files only, or both with one generated from the other?
19. Should client contactability status be stored as a compact current-state block, or derived only from activity logs at runtime?
20. Should demand readiness be a primary file under `demand/readiness.md`, or should it be generated from provider, lien, claim, and document ledgers every time?
21. Should demand drafts/finals live under `demand/`, `documents/generated/`, or both with `demand/` acting as the workflow view and `documents/` as the document library?
22. Should offer history live entirely inside each claim ledger, or should each offer also get its own first-class file under `negotiation/offers/`?
23. Should offer analyses be separate durable files even when the recommendation is simple, or only when an attorney/client decision is required?
24. Should client settlement authority be a structured block in the claim ledger, a separate privileged activity entry, or both?
25. Should negotiation summaries be generated views from claim ledgers, or manually curated attorney-facing summaries?
26. How should confidential attorney strategy notes be represented if the vault is a masked operational shadow that may later be productized for multi-firm use?
27. Should settlement execution live in a single `settlement/settlement.md` ledger, or split across statement, release, trust, and distribution files?
28. How much trust-account information should the masked vault store versus linking to the real accounting system or scanned source records?
29. Should release review be a required attorney-review workflow for every settlement, or configurable by claim type/value/carrier?
30. Should unresolved liens move the whole case into a separate lien phase, or can the case be "partially closed" with a trust holdback and open lien workflow?
31. Should supplemental settlement statements live under `settlement/`, `liens/`, or both?
32. Should lien final-resolution status vocabulary be standardized exactly as `outstanding/final_requested/final_received/negotiated/paid/satisfied/waived`, or does the firm need additional legal statuses?
33. Should `all_liens_resolved` require satisfaction/release documents for every paid lien, or is payment documentation enough for some lien types?
34. Should the final distribution workflow require a client signature on the supplemental statement, or only delivery plus trust-account documentation?
35. Should lien-phase workflows live under a separate `liens/` UI section, the `settlement/` UI section, or both?
36. What exact closure type vocabulary should the data contract use: settled, litigated, declined, withdrawn, dismissed, transferred, duplicate, other?
37. Should `all_obligations_complete` be a derived field only, or may an attorney manually override it with a required reason?
38. What retention periods should be encoded for minors, litigation files, declined intakes, and withdrawn cases?
39. Should reopened matters always become new linked cases, or are there limited cases where the same case can return to active?
40. Should litigation have a single `litigation/litigation.md` master ledger plus subfolders, or should each subphase own its own primary ledger?
41. How should court scheduling orders become workflow timers: parsed automatically from filed orders, entered manually by staff, or both?
42. Should discovery completion be derived only from discovery/deposition records, or can an attorney mark discovery substantially complete to trigger mediation?
43. How should appeal/post-judgment workflows fit after trial: separate phase, litigation subphase, or closure branch?
44. Should settlement during litigation route to the same settlement workflow with a litigation fee-rate modifier, or should it have a distinct litigation-settlement workflow?
45. Should defendant records live in a shared `litigation/parties.md`, inside `service.md`, or as one file per defendant?
46. Should complaint template selection be stored as attorney-reviewed metadata before drafting, or only recorded after the complaint is drafted?
47. Should discovery eligibility be tracked per defendant, or is there any firm preference to wait until all defendants are served/answered before launching discovery?
48. How should motions to dismiss fit into 7.1: answer-processing branch, separate motion practice workflow, or both?
49. Should each written discovery set be a separate markdown file, or should all sets live in one `litigation/discovery/discovery.md` ledger with linked document shadows?
50. Should RFA timing be represented as a workflow decision node, since RFAs may be strategic rather than automatic with initial discovery?
51. What level of deficiency severity should block mediation: only high severity, attorney-marked material deficiencies, or any unresolved deficiency?
52. Should motion-to-compel work become its own generic motion-practice workflow rather than living inside written discovery?
53. Should every deposition event be a separate file, or should low-value witness depositions live only in a shared deposition ledger?
54. Should the deposition schedule be a generated view from deposition events, or a manually editable file that synchronizes back into events?
55. What default rule decides which depositions are required before mediation: attorney plan, workflow recommendation, scheduling order, or case type?
56. Should testimony trackers be one case-wide file, one file per deposition, or both with a generated aggregate?
57. Should records-only third-party subpoenas use the deposition workflow or a separate subpoena/records workflow?
58. Should mediation authority amounts be stored in the masked vault at all, or should the vault only record that authority was confirmed with a pointer to privileged storage?
59. Should each mediation event have its own folder, or should repeated/continued mediation sessions live in one event file with multiple sessions?
60. How should partial settlements be modeled when one defendant or claim resolves but other litigation tracks remain active?
61. Should mediator proposals be represented as timer nodes inside mediation, or as a separate settlement-offer workflow with a proposal response deadline?
62. Should the scheduling order be parsed into a dedicated deadline ledger automatically, or should trial deadlines be entered manually after attorney/staff review?
63. Should treating physicians be represented as expert records for trial-prep purposes even when they are not retained experts?
64. Should exhibit and witness lists be generated views from source records, manually curated filings, or both?
65. How should privileged trial strategy, motions in limine strategy, and impeachment plans be represented without exposing more than the masked vault should hold?
66. Should trial-ready review require attorney approval every time, even when all checklist predicates are true?
67. How detailed should live trial daily notes be in the masked vault, given that much of trial strategy may be privileged?
68. Should admitted exhibits be tracked separately from the filed exhibit list, or should the exhibit list record have admitted/excluded status fields?
69. Should plaintiff verdict collection use the existing settlement workflow with judgment-specific fields, or should judgment collection be a separate workflow?
70. Should defense verdict always trigger an attorney appeal/post-trial review task before closure?
71. How should mistrial be represented: a litigation continuation state, a new trial workflow instance, or a closure/settlement decision gateway?
