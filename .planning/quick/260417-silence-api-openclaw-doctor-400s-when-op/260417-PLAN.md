<objective>
Stop the /api/openclaw/doctor endpoint from flooding the browser console with 400 errors
when OpenClaw isn't installed. The dashboard polls this endpoint on an interval; each
failed poll surfaces a red "Failed to load resource: 400" in DevTools even though the
"OpenClaw is not installed" condition is an expected, benign state for users who haven't
onboarded the runtime yet.

Purpose: Improve UX for users running Mission Control without OpenClaw. No console spam,
no misleading error noise, while keeping the onboarding flow (runtime-setup-modal) and
the health banner behaving correctly.

Output:
- GET handler returns 200 with a soft `{ installed: false, healthy: false, ... }` payload
  when the openclaw CLI is missing (ENOENT / "not installed" / "not reachable" / "command
  not found").
- GET handler continues returning 5xx for genuine server errors (no swallowing real bugs).
- POST handler is intentionally UNCHANGED: it still returns 400 on missing CLI because
  POST is an explicit user action ("Run doctor fix") and deserves an error surface.
- Banner (`OpenClawDoctorBanner`) gains a one-line guard so it doesn't render a
  "not installed" warning — the onboarding modal owns that case.
- No changes to `parseOpenClawDoctorOutput` (the parser is unaware of install state).
- No test changes: the repo has no test asserting the 400 on ENOENT (verified:
  `src/lib/__tests__/openclaw-doctor.test.ts` tests the parser only;
  `tests/auth-guards.spec.ts` only checks the endpoint is auth-protected, not status codes;
  no `__tests__` directory exists under `src/app/api/openclaw/doctor/`).
</objective>

<decision_log>
Decisions locked by the user at plan time:

1. **GET becomes soft 200 when CLI is missing.** The polling GET is the source of the
   console-spam problem. Changing only the "missing CLI" branch (already guarded by
   `isMissingOpenClaw(detail)`) to 200 is the minimal fix.

2. **POST stays at 400 when CLI is missing.** POST only runs when the user clicks
   "Run Doctor Fix" or "Auto-Fix Issues" — an explicit, user-initiated action. A 4xx there
   is the correct surface (it will display the error in the banner/modal, which is what
   we want for that intentional click). This mirrors the user's constraint in the brief.

3. **Payload shape: extend `OpenClawDoctorStatus` with an optional `installed` field.**
   Adding `installed?: boolean` is non-breaking — existing consumers that don't check it
   continue to work. The "missing CLI" 200 response sets `installed: false`; the healthy
   path sets `installed: true` (or omits — consumers default to "installed" semantics when
   the field is absent, but we set it explicitly for clarity).

4. **Client discriminator in the banner only.** The banner gets a one-line early return
   when `doctor.installed === false`. The onboarding modal (`runtime-setup-modal.tsx`)
   is intentionally left unchanged:
   - The modal's `checkHealth` / `runOnboard` today silently fails when GET returns 400
     (the `if (doctorRes.ok)` branch skips). After the change, the modal will receive the
     soft payload with `healthy: false, summary: "OpenClaw is not installed"` and advance
     to the `verify` step showing that summary. That is a **net UX improvement** over
     today's silent failure, so we leave it.
   - `runDoctorFix` in the modal calls POST, which still 400s on missing CLI — unchanged
     behavior.

5. **Error-branch behavior for non-ENOENT failures is preserved.** The existing fall-
   through in the GET catch block (passing `detail` / `code` through
   `parseOpenClawDoctorOutput` and returning 200 with a warning/error level) stays
   exactly as-is. Only the `isMissingOpenClaw(detail)` sub-branch is converted from 400
   to a soft-200 "not installed" payload.
</decision_log>

openclaw CLI" branch from 400 to a soft 200. Do NOT touch the POST handler.

Specifically:

1. In the GET `catch` block (around line 42-45 today), replace:
```ts
if (isMissingOpenClaw(detail)) {
  return NextResponse.json({ error: 'OpenClaw is not installed or not reachable' }, { status: 400 })
}
```

with a 200 response that has the shape consumers expect (mirrors
`OpenClawDoctorStatus` from `src/lib/openclaw-doctor.ts`), plus an `installed: false`
discriminator. Use inline object literal (do NOT import/extend `OpenClawDoctorStatus`
type in the route — keep it a plain JSON payload; the type compatibility is by shape):

```ts
if (isMissingOpenClaw(detail)) {
  return NextResponse.json(
    {
      installed: false,
      level: 'warning' as const,
      category: 'general' as const,
      healthy: false,
      summary: 'OpenClaw is not installed or not reachable',
      issues: [],
      canFix: false,
      raw: detail,
    },
    {
      status: 200,
      headers: { 'Cache-Control': 'no-store' },
    }
  )
}
```

Key requirements:
- `status: 200` (not 400) — this is the whole point of the change.
- `healthy: false` — it's not healthy, but it's also not a *doctor* warning; the banner
  will key off `installed: false` to suppress rendering.
- `canFix: false` — "install the CLI" is not something the doctor --fix POST can do.
- `raw: detail` — preserves the underlying spawn error text (useful for DevTools
  inspection and matches the existing pattern elsewhere in the file).
- Keep `'Cache-Control': 'no-store'` — consistent with the other GET responses.

2. Add an `installed: true` field to the two non-error GET response payloads so
   consumers can rely on the discriminator being present. Wrap the `parseOpenClawDoctorOutput`
   result with a spread:

   **First GET 200 response (try-block, around line 36):**
```ts
return NextResponse.json(
  {
    installed: true,
    ...parseOpenClawDoctorOutput(`${result.stdout}\n${result.stderr}`, result.code ?? 0, {
      stateDir: config.openclawStateDir,
    }),
  },
  { headers: { 'Cache-Control': 'no-store' } }
)
```

   **Second GET 200 response (catch-block fall-through for non-ENOENT errors, around line 47):**

```ts
  {
    installed: true,
    ...parseOpenClawDoctorOutput(detail, code ?? 1, {
      stateDir: config.openclawStateDir,
    }),
  },
  { headers: { 'Cache-Control': 'no-store' } }
)
```

3. POST handler: leave completely unchanged. Do not modify the two `isMissingOpenClaw`
   branches in POST (they should continue to return 400 — explicit user action).

4. Do not modify `isMissingOpenClaw`, `getCommandDetail`, or imports.

5. Genuine server errors (anything that bypasses the `try` and the `isMissingOpenClaw`

   throws outside the handler (unhandled rejections, etc.), which is the Next.js default.
