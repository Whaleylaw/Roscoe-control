<objective>
Build the **blocker flow + resume path + runner-exit emission** on top of the Wave 1 substrate. This plan closes CP-03, CP-04, the SCHED-05 third emission point, and the SCHED-06 `task.container_exited` / enriched `task.checkpoint_added` broadcasts.

End-to-end blocker flow (per 15-CONTEXT.md § Blocker & Resume Flow — LOCKED):
1. Agent POSTs `/api/tasks/:id/checkpoints` with `status: 'blocked', blocker_reason: '...'`.
2. POST handler (extended here from Plan 15-04): INSIDE the same db.transaction that INSERTs task_checkpoints, additionally UPDATEs `tasks SET status='awaiting_owner'` + INSERT comment (author='system'). The transaction now has 3 DB ops (plus JSONL append). Rollback on any failure.
3. Broadcast `task.checkpoint_added` enriched with `blocker_reason` in the payload (so the daemon's SSE handler sees it without re-querying).
4. Daemon (`scripts/mc-runner.mjs`) SSE handler for `task.checkpoint_added`: when `status==='blocked'` AND `activeTasks.has(task_id)`, call `docker stop --time=15 <containerId>`. Worktree + `.mc/` preserved.
5. Container exits (clean SIGTERM or SIGKILL after 15s). Daemon's `watchContainerExit` fires → `postRunnerExit` POSTs `/api/runner/tasks/:id/runner-exit`.
6. runner-exit handler (extended here): detects the task is already in `awaiting_owner` status → emits `task.container_exited { reason: 'blocked' }` (overrides the normal exit reason). Attempt row persisted as usual. Status NOT flipped again.

Resume path (CP-04 via existing Phase 14 mechanics):
7. Owner moves the task `awaiting_owner → assigned` via PUT /api/tasks/:id (existing Phase 10 transition path). The existing task.status_changed broadcast already fires; we add `task.runner_requested` re-emission on this specific transition.
8. Scheduler's 30s tick (Plan 15-02) picks up the assigned recipe-task via autoRouteInboxTasks (actually no — it's already assigned, not inbox). The runner-requested broadcast from step 7 IS what the daemon hears. Daemon claims.
9. Claim route (`src/app/api/runner/claim/[task_id]/route.ts`) queries the most recent blocked checkpoint for this task. If one exists AND the task has `runner_attempts > 0`, constructs `resume_marker: { blocker_reason, at_iso }` and includes it in the dispatch payload.
10. Daemon receives dispatch payload → passes `resume_marker` into `seedMcDir(worktreePath, { task, resume_marker })`. Plan 15-03 delivered the seedMcDir extension; this plan wires the consumer.

Non-blocker exit emission (SCHED-06 part):
- runner-exit handler ALSO broadcasts `task.container_exited` for every exit (timeout / crash / exit:N / worktree_create_failed). `reason` = the runner-reported reason. For `awaiting_owner` pre-flip → reason='blocked'.
- runner-exit retry path (existing Phase 14 logic: when exit is retriable AND attempts < max, task flips back to assigned) now ALSO emits `task.runner_requested` — this is the THIRD SCHED-05 emission point (completes the set; autoRouteInboxTasks + POST /api/tasks already covered in Plan 15-02).

    ```ts
     * checkpoints after an owner flipped the task BACK and BEFORE it got
     * re-claimed, we still want the blocker marker (the agent's follow-ups
     * won't make the marker misleading — they're context for the next attempt).
     *
     * Rule: inject ONLY when the LATEST checkpoint is status='blocked'.
     */
    export function resolveResumeMarker(db: Database.Database, taskId: number): ResumeMarker | null {
      const row = db.prepare(`
        SELECT status, blocker_reason, created_at
        FROM task_checkpoints
        WHERE task_id = ?
        ORDER BY id DESC
        LIMIT 1
      `).get(taskId) as { status: string; blocker_reason: string | null; created_at: number } | undefined

      if (!row) return null
      if (row.status !== 'blocked') return null
      if (!row.blocker_reason) return null

      return {
        blocker_reason: row.blocker_reason,
        at_iso: new Date(row.created_at * 1000).toISOString(),
      }
    }
    ```
