> **Doc-drift correction (Phase 18-03 / audit-td-3):** Original plan text described the agent `/submit` endpoint as a "terminal-flip to done" that sets `task.status='done'`. Per Phase 17-01 RTEST-02 the shipped implementation flips `in_progress → review` (NOT `done`); Aegis quality approval then flips `review → done` in a separate transaction. The Phase-14-era `{status: 'done'}` body shape is preserved in code blocks below for historical accuracy, but the shipped server transaction ignores the body value and always flips to `review`. Prose below has been corrected; code, tests, and schema unchanged by this correction.

<objective>
Ship three small runner-facing endpoints missing from the v1 Phase 14 plan but required by downstream plans:

1. **POST /api/runner/tasks/:task_id/submit** (runner-token) — the agent container's terminal-flip path. Hello-world agent (Plan 14-09) and the preamble forward-reference (Plan 14-07) both need this. Without it, the PUT to /api/tasks/:id that was originally drafted would fail the runner-token allowlist guard (only six /api/runner/tasks/:id/* paths are whitelisted for runner-tokens in src/lib/runner-tokens.ts — and /submit/ is already one of them, locked in Phase 11).

2. **POST /api/runner/tasks/:task_id/container-started** (runner-secret) — the daemon's "real container_id is now known" signal. Claim route (Plan 14-05) writes a `pending:<task>:<attempt>` placeholder; reconciliation (Plan 14-08b) must match against real docker container_ids. Without this handoff, the placeholder persists and reconciliation is broken.

3. **GET /api/runner/config** (runner-secret) — daemon's one-shot settings read. Runner daemon (Plan 14-08b) needs `runtime.project_repo_map` + `max_memory_per_container` + `max_cpu_per_container` + `failed_gc_window_days` + `max_concurrent_containers`. Consolidating into one endpoint avoids five settings round-trips at startup and is cleaner than exposing /api/settings to the operator role.

Purpose: Resolve three checker blockers in a single focused plan without re-splicing 14-04. Wave 1 so 14-08b can depend on all of them.
Output: Three routes + three test suites. All runner-secret or runner-token scoped (no session/cookie auth).
</objective>
