## Deviations from Plan

The plan's Task 2 preferred option was **B.b** — a single `test:docs` package.json entry with no config edits. In execution this turned out to be mechanically impossible: vitest 2.1 rejects `--include` as a CLI flag, and the default `vitest.config.ts` `include` pattern (`src/**/*.test.ts`) does not match `scripts/__tests__/**/*.test.mjs`. The only routes to run the harness tests were (a) edit `vitest.config.ts`, (b) add a new config file, or (c) swap to `node --test`.

Chosen: **(b) — add `vitest.docs.config.mjs`**, a 14-line minimal config. This was the smallest-blast-radius option — the primary `vitest.config.ts` is untouched, product tests (`pnpm test`) retain their original surface area, and the harness tests get a first-class run surface via `pnpm test:docs`. The plan's `files_modified` lists 3 files; this makes it 4. The deviation is surfaced here and reflected in the `key-files.created` frontmatter block.

### Auto-fixed Issues (harness-surfaced drift)

The harness's first run against the completed Phase 18.1 doc set surfaced three real drift hits. All three were fixed inline per the plan's explicit directive: *"If it fails on the current tree, DO NOT loosen assertions — fix the drift in the offending doc (the harness is the authority)."*

**1. [Rule 1 — Bug] Broken link to runner-docker.test.ts in runner-daemon.md**
- **Found during:** Task 1 verification (first `pnpm docs:verify-runtime` run)
- **Issue:** `docs/runtime/runner-daemon.md:167` linked to `src/lib/__tests__/runner-docker.test.ts`; the actual file is `runner-docker-args.test.ts` (already correctly cited elsewhere in `agent-contract.md:93`).
- **Fix:** Changed the path in the blockquote.
- **Commit:** `913d188`

**2. [Rule 1 — Bug] Same broken link in admin-config.md**
- **Found during:** Task 1 verification
- **Issue:** `docs/runtime/admin-config.md:151` had the same broken path.
- **Fix:** Changed the path in the blockquote.
- **Commit:** `913d188`

**3. [Rule 1 — Bug] INDEX.md contained the literal `submit→done` pattern**
- **Found during:** Task 1 verification
- **Issue:** `docs/runtime/INDEX.md:62` described the DESIGN-ERA spec's drift using the literal phrase `(submit→done, ro-overlay workspace mode, …)` — the drift-guard caught it because the pattern is unscoped-by-design (any occurrence fails, even when the doc is explicitly framing the term as legacy).
- **Fix:** Rephrased to "the legacy direct-to-done submit lifecycle" — same meaning, no literal-pattern hit.
- **Commit:** `913d188`

### Harness-correctness fixes (not drift in docs — bugs in the harness itself)

During the first-run triage I corrected two bugs in the harness I had just authored, before committing it:

- **Slugify mishandled consecutive whitespace.** Initial implementation collapsed `\s+` to a single dash, producing `reading-order-preamble-soul-mc` for a heading whose real GitHub slug is `reading-order-preamble--soul--mc`. Fixed by preserving whitespace during punctuation removal and replacing each single space with a single dash. The Vitest suite now includes two tests pinning this behavior.
- **JSON fence parsing rejected legitimate JSONL examples.** `getting-started.md:135` has a three-line boot-log fence (three separate JSON objects, one per line). Added `tryParseJsonOrJsonl()` that parses single-object first, falls back to line-by-line JSONL; fails only if NEITHER shape parses.

Both were fixed before the harness was committed (`55130ab`), so the harness shipped correct on first commit.
