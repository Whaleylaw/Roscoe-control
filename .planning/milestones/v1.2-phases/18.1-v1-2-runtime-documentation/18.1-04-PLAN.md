## Section: RecipeBadge
- Where it renders: Every task card on the Kanban when `task.recipe_slug` is set (RUI-01)
- Visible state: Recipe name (from IndexedRecipe cache) + a colored dot keyed to the recipe's `model.primary` tier (Opus=high / Sonnet=mid / Haiku=low / unknown=neutral muted)
- Cache-miss fallback: Shows the raw `recipe_slug` literal (NOT null) during first paint / SSE reconnect windows
- data-testid: `recipe-badge` (added Phase 18-02 commit `96c57d9` for E2E locator stability)
- Cite: `src/components/panels/task-card/recipe-badge.tsx`, `src/lib/model-tier-colors.ts` (MODEL_TIER_COLORS + modelTierClassName helper)
- Consumes: Zustand recipes slice (seeded via useServerEvents on mount; updates via `mc:recipe-indexed` / `mc:recipe-removed` CustomEvents)

## Section: RunnerStatusBanner
- Where it renders: Inside task-board-panel.tsx at line ~1001 (between the error region and the Kanban grid). NOT in the global header — this is scoped to the task-board view only.
- Visible states (3 render branches):
  - Loading: null-render (no mount flicker) while first fetch is in flight
  - OK: `🟢 Runner online` (no waiting tasks) OR `🔴 Runner offline — tasks waiting: N` (when offline with queued recipe-tagged tasks)
  - Error: muted "Runner status unavailable" fallback (never blocks the board, never throws)
- How "online" is computed: The `/api/runtime/runner-status` endpoint returns `online: true` if any heartbeat row has `last_heartbeat_at` within `STALE_WINDOW_SECS=90` (90 seconds).
- Accessibility: `role="status" aria-live="polite"` on wrapper — state transitions announce to screen readers without stealing focus.
- Polling cadence: `POLL_INTERVAL_MS=10_000` (10s). Three SSE event types (`mc:task-container-started`, `mc:task-container-exited`, `mc:task-runner-requested`) trigger a `REFRESH_DEBOUNCE_MS=1_000` re-fetch.
- Waiting count: Workspace-scoped per `auth.user.workspace_id` (viewer-tier endpoint).
- EXPLICIT PITFALL CALLOUT #9 — "⚠️ `🟢 Runner online` means heartbeat fresh within 90s, NOT that Docker is reachable right now":
  - "A runner that exits code 2 (Docker unreachable) and is respawned by LaunchAgent every 30s will still UPSERT a heartbeat on each boot — remaining 'online' during the 90s stale window. The banner does not inspect Docker health; it only inspects heartbeat freshness."
  - "If tasks are stuck in `assigned` despite a green banner, check `.data/runner/daemon.err` for exit code 2 and cross-reference [`scripts/README.runner.md#troubleshooting`](../../scripts/README.runner.md#troubleshooting)."
- Cite `src/app/api/runtime/runner-status/route.ts` (STALE_WINDOW_SECS, fresh-heartbeat query) + `src/components/panels/runner-status-banner.tsx` (the three render branches).

## Section: Progress tab
- Where it renders: Inside the task detail view, new tab added by Phase 16 (RUI-03)
- Tab label source: `progressT('tabLabel')` against `taskBoard.progressTab.tabLabel` i18n namespace (seeded across 10 locales in Phase 16-01)
- Visible content: Live checkpoint timeline grouped by attempt (newest attempt first). Each attempt row lists its checkpoints with step, status, summary, artifacts, tokens_used, duration_ms. Blocker checkpoints show the blocker_reason prominently.
- Empty state: `taskBoard.progressTab.empty` i18n key
- Error state: `taskBoard.progressTab.loadError`
- Live updates: Consumes `GET /api/tasks/:id/checkpoints` on mount + `mc:task-checkpoint-added` CustomEvents for real-time merging.
- De-dupe strategy: Uses `Map<number, Checkpoint>` keyed by checkpoint id — handles SSE-replays and fetch-after-mount overlap without sort-on-every-push.
- Auto-scroll: Smooth-scrolls to top (newest-first) when checkpoint count increases UNLESS the user has scrolled down (>16px from top). Returns to anchored-top when user scrolls back up.
- Artifact types (CP-05): `file | url | diff | test_result | comment | other` — each rendered with type-specific icon and optional path/url/ref/summary.
- Cross-reference lifecycle: When the Progress tab shows an attempt that ended with "submit," the task will transition `in_progress → review → done` (two-hop via runAegisReviews). Progress tab does NOT show the review→done transition itself — that is a `task.status_changed` event reflected on the Kanban card. Cross-link to `docs/runtime/agent-contract.md#submit-http-endpoint`.
- Cite: `src/components/panels/task-detail/progress-tab.tsx`.

## Section: RecipeCombobox
- Where it renders: Create task form + Edit task modal — the Recipe dropdown (RUI-04)
- Behavior: Autocomplete via `GET /api/recipes/search?q=...` — 300ms debounce on input + AbortController cleanup on unmount/refire. Rapid typing aborts in-flight requests.
- NO client-side result caching — always debounced fetch per Phase 16-05 anti-pattern rule.
- Read-only consumer of Zustand recipes slice (owned by Plan 16-02). Uses defensive selector: `(s as {recipes?: Recipe[]}).recipes` with slug-literal fallback on pre-hydration.
- Cite: `src/components/panels/task-form/recipe-combobox.tsx`.

## Section: Advanced section (mounts + skills + model override)
- Location: Collapsed by default in Create / Edit task forms. Opens `read_only_mounts`, `extra_skills`, and `model_override` editors.
- `read_only_mounts`: Editor accepts rows of `{source, mount_as, label}`. Source must be prefix-match of `runtime.mount_allowlist` entries — server rejects otherwise. Per-row errors from 400 responses (issues[] aggregated validation) mapped via regex `^read_only_mounts\.(\d+)\.` into MountsEditor's errors prop.
- `extra_skills`: Simple name list; bounded by `runtime.extra_skills_cap` (default 20). Each skill mounted at `/skills/<name>` ro.
- `model_override`: Dropdown of registry models. If set, overrides `recipe.model.primary` at claim time.
- Cite `src/components/panels/task-form/` (the Advanced section lives inside the task-form tree).

## Section: Recipes panel
- Where: Main nav -> "Recipes" (priority=false, essential=false — a full-mode inspection surface, not in Essential mode)
- Renders: Indexed recipe list with name, description, model, tags, status (valid / error).
- Admin action: "Resync" button triggers `POST /api/recipes/resync` and shows an inline feedback banner (github-sync-panel precedent; NOT a toast library).
- Per-row View toggle: Expands soul_md inline via shared MarkdownRenderer (no modal, no extra route).
- Error rows surface the `error_message` column (NOT `indexed_error` — renamed in Phase 18-04 commit `d42983a`).
- Cite: `src/components/panels/recipes-panel.tsx`, `src/app/api/recipes/resync/route.ts`.

## Section: Blocker-flow UX
- When an agent posts a checkpoint with `status:'blocked'`, the task transitions to `awaiting_owner`. The UI:
  - Shows the task on the Kanban with an "Awaiting owner" status (not in_progress, not review)
  - Surfaces an auto-posted system comment containing `blocker_reason`
  - The Progress tab highlights the last checkpoint with a red blocker badge
- Resolve path: Operator reads the blocker_reason, fixes the underlying dependency (e.g. adds a missing secret file), and manually moves the task back to `assigned` (or uses a future UI resolve action). The runner's SSE handler detects the `assigned` re-entry and launches attempt N+1 with `is_resuming=true`. The runner-injected resume marker is appended verbatim to `.mc/progress.md` (format LOCKED Phase 17-05).
- Cite: `src/app/api/tasks/[id]/checkpoints/route.ts` (CP-03 blocker branch), `src/lib/task-dispatch.ts` (resume routing).

## Section: RECIPE_LOCKED client-side gate
- Formula: `isDispatched = task.status !== 'inbox' && task.status !== 'assigned'`
- Effect: When `isDispatched` is true, EditTaskModal disables `RecipeCombobox` + the Advanced section and skips runtime-field delta on the PATCH payload. An in-progress / review / done / failed / awaiting_owner task cannot change its recipe, read_only_mounts, extra_skills, or model_override.
- Rationale: Those fields drive container composition at claim time; changing them after claim would leak misaligned state.
- Cite: Phase 16-05 LOCK.

## Section: Event wiring
- 3-column table "Zustand / CustomEvent name | Emitted by | Consumed by":
  - `mc:recipe-indexed` | server SSE broadcast from recipe-indexer | Zustand recipes slice refresh
  - `mc:recipe-removed` | server SSE | Zustand recipes slice delete
  - `mc:task-checkpoint-added` | server SSE from checkpoints route | ProgressTab merge + RunnerStatusBanner debounced re-fetch
  - `mc:task-container-started` | server SSE from container-started route | RunnerStatusBanner debounced re-fetch
  - `mc:task-container-exited` | server SSE from runner-exit route | RunnerStatusBanner debounced re-fetch
  - `mc:task-runner-requested` | server SSE from autoRouteInbox + POST /api/tasks + runner-exit retry | RunnerStatusBanner debounced re-fetch
- Cite: `src/lib/use-server-events.ts:152-158` (chat.message precedent pattern), STATE.md Phase 16-01 lock.
