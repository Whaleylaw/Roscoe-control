Structure:

## Header
- H1: "Getting Started with Recipe Agents"
- Short paragraph: "This tutorial walks you from a fresh Mission Control install to your first recipe-tagged task completing via the ephemeral runner. Expect 10 minutes of wall-clock time on a Docker-equipped machine."
- Prerequisites block (bullets): Node 22+, Docker Desktop running, pnpm available, a local git repo you control
- Pointer block: "For deep reference on any step, see the links at the end of this tutorial."

## Step 1: Install and boot Mission Control
- Commands:
  ```bash
  git clone <your-mc-repo>
  cd mission-control
  pnpm install
  pnpm build
  pnpm dev          # default localhost:3000
  ```
- Expected: Visit `http://localhost:3000/setup` — if no admin exists, create one.
- Note: For standalone mode (`node .next/standalone/server.js`), you MUST set `MISSION_CONTROL_RECIPES_DIR=$(pwd)/recipes` — see [admin-config.md](./admin-config.md#standalone-mode-requirements) (pitfall 7 cross-link).

## Step 2: Export MC_URL and MC_API_KEY
- Commands:
  ```bash
  export MC_URL=http://127.0.0.1:3000
  export MC_API_KEY=$(cat .data/.auto-generated | grep API_KEY= | cut -d= -f2-)
  curl -s -H "Authorization: Bearer $MC_API_KEY" "$MC_URL/api/status" | jq .
  ```
- Expected: JSON status response.
- Note: The API key is auto-generated on first boot and persists to `.data/.auto-generated`. See [admin-config.md#auth-tiers--secrets](./admin-config.md#auth-tiers--secrets).

## Step 3: Configure admin settings
- EXPLICIT PITFALL #4 inline warning: "⚠️ `runtime.project_repo_map` is the ONLY way the runner learns where your repos live. There is no env-var fallback."
- Commands:
  ```bash
  # Allow the runner to mount any path below ~/Github (adjust to your filesystem)
  pnpm mc settings set runtime.mount_allowlist '["/Users/me/Github"]'

  # Tell the runner project_id 1 lives at this absolute path
  pnpm mc settings set runtime.project_repo_map '{"1":"/Users/me/Github/my-repo"}'
  ```
- Verify:
  ```bash
  curl -s -H "Authorization: Bearer $(cat .data/runner.secret)" "$MC_URL/api/runner/config" | jq .
  ```
- Expected: JSON with `project_repo_map`, `max_concurrent_containers`, `failed_gc_window_days`, `mount_allowlist`.

## Step 4: Build the reference image
- Commands:
  ```bash
  pnpm mc:build-hello-world
  # OR equivalently:
  bash docker/hello-world-agent/build.sh
  ```
- Expected: `Successfully tagged mc-hello-world-agent:latest`
- Verify: `docker images mc-hello-world-agent`
- See [docker/hello-world-agent/README.md](../../docker/hello-world-agent/README.md) for the 7-step contract the image implements.

## Step 5: Start the runner daemon (separate terminal)
- Commands:
  ```bash
  # In a second terminal, from the repo root:
  node scripts/mc-runner.mjs
  ```
- Expected: JSON log lines including `"msg":"config loaded"`, `"msg":"runner boot"`, `"msg":"reconcile"`.
- Leave this terminal open — the runner polls for work.
- If the runner exits 1: your `.data/runner.secret` is missing or `/api/runner/config` is unreachable. If it exits 2: Docker is down. See [runner-daemon.md#exit-codes](./runner-daemon.md#exit-codes).

## Step 6: Create a recipe-tagged task
- Commands:
  ```bash
  # Create a task with recipe_slug pointing at the reference recipe
  TASK=$(curl -s -X POST "$MC_URL/api/tasks" \
    -H "Authorization: Bearer $MC_API_KEY" \
    -H "Content-Type: application/json" \
    -d '{
      "title": "Hello world smoke",
      "recipe_slug": "hello-world",
      "workspace_source": {"project_id": 1, "base_ref": "main"}
    }')
  TASK_ID=$(echo "$TASK" | jq -r .id)
  echo "Created task $TASK_ID"
  ```
- Expected: A task id is printed. The task's initial status is either `inbox` (then autoRouted to `assigned`) or directly `assigned` if the workspace config permits — see [task-board-surfaces.md#event-wiring](./task-board-surfaces.md#event-wiring).

## Step 7: Watch the task progress
- Option A: UI observation. Open `http://localhost:3000`, click into the task, open the Progress tab. You will see:
  - A checkpoint row for `hello-world-smoke` appear within a few seconds
  - The task card's status transition `assigned → in_progress`
  - After the container exits and submits, the task transitions to `review` (NOT directly to `done`)
  - After Aegis review (automatic, runs as part of the scheduler), the task flips `review → done`
- EXPLICIT PITFALL #1 inline note: "⚠️ Submit flips the task to `review`, not `done`. The terminal `review → done` hop is performed by Aegis's automated review pass. If your task stops at `review`, check the review comments on the task. See [agent-contract.md#submit-http-endpoint](./agent-contract.md#submit-http-endpoint)."
- Option B: CLI observation.
  ```bash
  pnpm mc events watch --types task
  # Expect lines like:
  # task.runner_requested {id: N, ...}
  # task.container_started {id: N, container_id: ...}
  # task.checkpoint_added {id: N, step: hello-world-smoke, status: completed}
  # task.container_exited {id: N, reason: exit, exit_code: 0}
  # task.status_changed {id: N, from: in_progress, to: review}
  # task.status_changed {id: N, from: review, to: done}
  ```
- Option C: Inspect the log stream for the running container:
  ```bash
  tail -f .data/runner/logs/task-$TASK_ID/latest/stderr.log
  ```

## Step 8: Verify the workspace artifacts
- The reference agent commits `HELLO.md` inside the task's worktree. Find it:
  ```bash
  # Worktrees live under .data/runner/worktrees/task-<id>/ while alive;
  # destroyed on 'done' (or retained for runtime.failed_gc_window_days on failure).
  # Inspect .mc progress + checkpoints via the log dir (preserved after task completion):
  cat .data/runner/logs/task-$TASK_ID/attempt-1/stderr.log | tail -20
  ```
- Expected: the stderr log shows the 6 agent steps + a successful submit.

## Troubleshooting
- A 6-row bullet list of the most common first-run issues and their remediation, each cross-linked to the appropriate reference doc:
  - Task stays in `inbox` → scheduler not running / recipe not indexed. Check Recipes panel for error row.
  - Task stays in `assigned` → runner offline. Check [runner-daemon.md#exit-codes](./runner-daemon.md#exit-codes).
  - Banner says 🟢 but nothing happens → the 90s stale-window gotcha; check `daemon.err` for exit code 2. See [task-board-surfaces.md#runnerstatusbanner](./task-board-surfaces.md#runnerstatusbanner).
  - Agent logs show `401` on submit → runner-token expired (task exceeded timeout). Cross-link.
  - Agent logs show 404 on `http://localhost:3000` → inside-container code used `localhost` instead of `$MC_API_URL`. Cross-link to [agent-contract.md#container-env-vars](./agent-contract.md#container-env-vars).
  - Recipe rejected by indexer → see [recipes.md#common-errors](./recipes.md#common-errors).
