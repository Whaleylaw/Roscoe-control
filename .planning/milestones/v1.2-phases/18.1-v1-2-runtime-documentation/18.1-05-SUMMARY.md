## Decisions Made

- **Numbered-step tutorial, not reference doc.** Used Pattern 4 from 18.1-RESEARCH.md ("operator journey end-to-end narrative"), not Pattern 1 ("surface-scoped reference"). Mirrors `docs/quickstart.md`'s shape so first-time operators follow a familiar template. The five Wave-1 docs are the references; this tutorial is the ladder to them.
- **Three observation channels in Step 7.** Covered UI (Progress tab), CLI (`pnpm mc events watch`), and log-file tail. Different operators trust different surfaces; naming all three ensures the lifecycle transitions are visible regardless of workflow preference.
- **Inline ⚠️ at Step 3 and Step 7.** Placed pitfall callouts adjacent to the operator actions they guard, not in a separate appendix. Pitfall #4 (exclusive `pnpm mc settings set`) lands before the commands that would otherwise tempt an env-var shortcut; pitfall #1 (two-hop) lands before the observation flow where operators would otherwise conclude `review` = broken.
- **Terminal Troubleshooting table.** Six rows at the bottom of the doc — symptom / cause / remediation-cross-link. This is the last-mile surface for first-run frustration. Each row names a specific observable state (UI message, log snippet, or blocked transition) so operators can match their failure to a row in seconds.
- **Exported `TASK_ID` in Step 6 with `$(jq -r '.task.id // .id')`.** Tolerates either `.task.id` (canonical) or `.id` (flat) response shapes — matches what `scripts/mc-runner-smoke.sh` does defensively. Subsequent steps all reference `$TASK_ID`, keeping the tutorial free of `<id>` placeholders.
- **Used `$(grep '^API_KEY=' .data/.auto-generated | cut -d= -f2-)`** for the MC_API_KEY export in Step 2 — same pattern as `scripts/mc-runner-smoke.sh:resolve_api_key()` so the tutorial and the harness agree on how to read the auto-generated key.
- **Did NOT touch `pnpm mc settings set` with an env-var alternative path.** The plan is explicit: NEVER env-var fallbacks. The tutorial uses `pnpm mc settings set` verbatim in Step 3 and the ⚠️ callout says so.
- **Standalone-mode note scoped to Step 1.** Users on `pnpm dev` (the default path) do not need to care about `MISSION_CONTROL_RECIPES_DIR`. The `> Note` block is explicit but does not block the happy path.
