## Decisions Made

- **Uniform subsection template.** All six UI subsections use the same four-part layout (What it is / Visible states / Data source / Operator signals) so operators can tab through surfaces and always know where to look. Establishes the template the remaining five Phase 18.1 docs (recipes.md, runner-daemon.md, agent-contract.md, admin-config.md, getting-started.md) should adopt.
- **Pitfall-callout format.** Used a `>` blockquote with a `⚠️` sigil for the pitfall-9 section. This is a visual escalation — a single formatting element that signals "this will bite you" — placed adjacent to the state it guards against (the `🟢 Runner online` branch description).
- **Enumerated the gate, didn't describe the inverse.** RECIPE_LOCKED gate section lists the five statuses that activate `isDispatched = true` explicitly (in_progress, review, done, failed, awaiting_owner) instead of relying on the operator to compute the inverse of the two-status not-equals. Saves a mental hop every time the doc is referenced.
- **Kept the doc to 349 lines.** Plan required ≥220; landed at 349 because the uniform six-surface structure plus the three full policy sections (blocker-flow, RECIPE_LOCKED, event-wiring) require breadth. 349 gives enough room for structural completeness without forcing an operator to page through prose to find the column they need.
- **Drift-guard compliance.** Doc uses `error_message` verbatim for the recipes-panel column name; never writes the literal `indexed_error`. The plan's automated verify includes `! grep -q "indexed_error"`; this was respected throughout.
