## Section: 5-minute architecture overview
- Prose explaining the overall pipeline in 2-3 paragraphs:
  - A `recipe` is authored as a directory under `recipes/<slug>/` containing a `recipe.yaml` and a `SOUL.md`. The server indexes recipes at boot + via chokidar + via admin resync.
  - A `task` can be tagged with `recipe_slug`, optional `read_only_mounts`, `extra_skills`, and `model_override`. When the task enters the `assigned` state, the scheduler emits `task.runner_requested`.
  - The runner daemon (`scripts/mc-runner.mjs`) polls + subscribes to `task.runner_requested`, atomically claims tasks via `POST /api/runner/claim/:id`, seeds `.mc/` inside a per-task git worktree, and launches a recipe-declared Docker container.
  - The container runs a tool-agnostic AGENT that reads `/recipe/PREAMBLE.md` + `/recipe/SOUL.md`, appends progress to `.mc/progress.md`, optionally POSTs checkpoints, and finally POSTs `submit {status:'done'}`. The server flips the task to `review`; Aegis approves to `done` (two-hop).
- Include ONE Mermaid sequence diagram (fenced ```mermaid block, text-reviewable + Git-diffable) showing:
  ```
  sequenceDiagram
    actor Operator
    participant MC as Mission Control
    participant Sched as Scheduler

    participant Docker

    MC->>Sched: autoRouteInbox (inbox→assigned)

    Aegis->>MC: approve
    Note over MC: task.status: review → done
    Runner->>MC: POST /api/runner/tasks/:id/runner-exit
    Runner->>Runner: GC tick destroys worktree
  ```

### Ephemeral Agent Runtime (v1.2)

Mission Control ships a recipe-based ephemeral agent runtime. A recipe authored under `recipes/<slug>/` declares a Docker image, mount layout, model, and timeouts. The `scripts/mc-runner.mjs` daemon claims recipe-tagged tasks, launches per-task containers against a git worktree, and surfaces live checkpoint progress in the UI.

- **Tool-agnostic agent contract** — any container honoring the HTTP + filesystem contract (env vars, preamble, submit endpoint) works; not coupled to any specific agent SDK
- **Per-task git worktrees** — each task runs in isolation; worktrees preserved on crash for seamless resume
- **Live Progress tab** — checkpoints stream via SSE; attempts grouped per task
- **Two-hop lifecycle** — agent submits → task enters `review` → Aegis approves → task flips to `done`

See [Runtime overview](docs/runtime/INDEX.md) to get started.
```
