<objective>
Ship the RTEST-04 Playwright E2E spec. Per D-03 LOCKED decision, this spec uses the HIGH-FIDELITY path: boot the MC test server, spawn the REAL runner daemon, create a recipe-tagged task, let the daemon claim it and run the real reference container, then assert the UI updates live.

Two DOM invariants asserted:
  1. **RUI-01 (recipe badge):** `[data-testid="recipe-badge"]` visible on the task card with the hello-world recipe name
  2. **RUI-03 (live Progress tab):** `[data-checkpoint-id]` rows appear in the Progress tab as container posts checkpoints via SSE (no page reload)

Decision D-03 rationale: Playwright spawning the real runner daemon tests the full integration path including SSE delivery, container lifecycle, and DOM reactivity. Higher flake surface, so the spec includes retry/timeout discipline and deterministic readiness probes.

Also extends `scripts/e2e-openclaw/start-e2e-server.mjs` with a gated `PHASE17_SPAWN_RUNNER=1` mode so the Playwright webServer block can orchestrate both the server AND the runner daemon from one entry point. Without this, Playwright's single webServer pattern can't manage two child processes reliably.

Purpose: Prove the UI reacts to real runtime events driven by a real container — the final assertion in the v1.2 milestone. This catches SSE delivery bugs, DOM update bugs, and recipe-badge/Progress-tab regressions that in-process tests miss.

Output: One new Playwright spec file + one modification to the E2E server bootstrap.
</objective>
