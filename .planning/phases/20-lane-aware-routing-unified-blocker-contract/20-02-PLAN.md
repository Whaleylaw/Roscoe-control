<objective>
Give the legacy (non-recipe) dispatch path a structured owner-intervention pause/resume contract that mirrors the recipe runner's `awaiting_owner` semantics, without touching retry/fail behavior. A caller PUTs `/api/tasks/:id` with `{status: 'awaiting_owner', blocker_reason, blocker_kind, resume_hint}` to pause; a caller PUTs `{status: 'assigned'}` to resume. Both transitions are atomic, persist the envelope in the existing `runner_last_failure_reason` column (no schema migration), and preserve the assigned agent through the pause.

Purpose: Deliver ROUTE-02 and COMPAT-03 from REQUIREMENTS.md. Pairs with Plan 20-01 (lane-aware routing) and Plan 20-03 (shared event shape) to complete Phase 20's "automatic unless blocker" loop for legacy tasks.

Output: An updated `updateTaskSchema` that accepts the three optional blocker fields with strict validation; a PUT handler branch that (a) rejects pauses missing required fields with 400, (b) rejects recipe-tagged tasks with 409 (directing callers to the checkpoint API), (c) commits the pause atomically, and (d) commits the resume atomically while clearing the envelope. A dedicated vitest route-handler file proves the happy path + every error path + retry/fail preservation.
</objective>

Layout:

```typescript
// LOCKED Phase 20 / ROUTE-02 — legacy blocker contract branch.
// Only fires when the PUT is targeting awaiting_owner pause OR awaiting_owner resume.
// Recipe-tagged tasks are rejected — they must use POST /api/tasks/:id/checkpoints.

if (normalizedStatus === 'awaiting_owner' || normalizedStatus === 'assigned') {
  const isRecipe = currentTask.recipe_slug != null && currentTask.recipe_slug !== ''

  // --- Pause branch ---------------------------------------------------------
  if (normalizedStatus === 'awaiting_owner' && currentTask.status === 'in_progress') {
    if (isRecipe) {
      return NextResponse.json(
        {
          error: 'Recipe-tagged tasks must use the checkpoints endpoint to pause. POST /api/tasks/:id/checkpoints with status=\"blocked\" and a blocker_reason.',
          code: 'RECIPE_BLOCKER_VIA_CHECKPOINTS',
        },
        { status: 409 },
      )
    }

    const missing: string[] = []
    if (!body.blocker_reason || !body.blocker_reason.trim()) missing.push('blocker_reason')
    if (!body.blocker_kind) missing.push('blocker_kind')
    if (!body.resume_hint || !body.resume_hint.trim()) missing.push('resume_hint')
    if (missing.length > 0) {
      return NextResponse.json(
        { error: `Blocker pause requires ${missing.join(', ')}`, code: 'BLOCKER_FIELDS_MISSING', missing },
        { status: 400 },
      )
    }

    const envelope = JSON.stringify({
      blocker_reason: body.blocker_reason!.trim(),
      blocker_kind: body.blocker_kind!,
      resume_hint: body.resume_hint!.trim(),
    })

    const run = db.transaction(() => {
      const upd = db.prepare(
        `UPDATE tasks
           SET status = 'awaiting_owner',
               runner_last_failure_reason = ?,
               updated_at = ?
         WHERE id = ? AND workspace_id = ? AND status = 'in_progress' AND recipe_slug IS NULL`,
      ).run(envelope, now, taskId, workspaceId)
      if (upd.changes === 0) {
        // Raced with another transition OR recipe_slug appeared — surface a 409.
        throw new Error('concurrent_transition')
      }
      // Future: shared event emission happens AFTER commit in Plan 20-03.
    })

    try {
      run()
    } catch (err) {
      if (err instanceof Error && err.message === 'concurrent_transition') {
        return NextResponse.json(
          { error: 'Task status changed during blocker transition; retry', code: 'CONCURRENT_TRANSITION' },
          { status: 409 },
        )
      }
      throw err
    }

    // Fetch the fresh row and return it — skip the generic fieldsToUpdate write path.
    const fresh = db.prepare('SELECT * FROM tasks WHERE id = ? AND workspace_id = ?').get(taskId, workspaceId)
    // …existing mapTaskRow + event broadcast pattern used elsewhere in the file.
    eventBus.broadcast('task.status_changed', {
      id: taskId,
      status: 'awaiting_owner',
      previous_status: 'in_progress',
      reason: 'blocker_pause_legacy',
      workspace_id: workspaceId,
    })
    // Mirror the existing generic-write-path broadcast so clients that listen
    // on `task.updated` still see this pause. Keep shape consistent with
    // line 695 of this file — `parsedTask` there; here the mapped fresh row.
    eventBus.broadcast('task.updated', mapTaskRow(fresh))
    // Plan 20-03 will add a third `task.blocker_transition` broadcast here,
    // after both broadcasts above.
    return NextResponse.json({ task: mapTaskRow(fresh) })
  }

  // --- Resume branch --------------------------------------------------------
  if (normalizedStatus === 'assigned' && currentTask.status === 'awaiting_owner' && !isRecipe) {
    // No required fields on resume — owner just clears the pause.
    const run = db.transaction(() => {
      const upd = db.prepare(
        `UPDATE tasks
           SET status = 'assigned',
               runner_last_failure_reason = NULL,
               updated_at = ?
         WHERE id = ? AND workspace_id = ? AND status = 'awaiting_owner' AND recipe_slug IS NULL`,
      ).run(now, taskId, workspaceId)
      if (upd.changes === 0) throw new Error('concurrent_transition')
    })

    try {
      run()
    } catch (err) {
      if (err instanceof Error && err.message === 'concurrent_transition') {
        return NextResponse.json(
          { error: 'Task status changed during resume; retry', code: 'CONCURRENT_TRANSITION' },
          { status: 409 },
        )
      }
      throw err
    }

    const fresh = db.prepare('SELECT * FROM tasks WHERE id = ? AND workspace_id = ?').get(taskId, workspaceId)
    eventBus.broadcast('task.status_changed', {
      id: taskId,
      status: 'assigned',
      previous_status: 'awaiting_owner',
      reason: 'blocker_resume_legacy',
      workspace_id: workspaceId,
    })
    // Mirror the existing generic-write-path `task.updated` broadcast so
    // `task.updated` subscribers still see this resume.
    eventBus.broadcast('task.updated', mapTaskRow(fresh))
    // Plan 20-03 will add a third `task.blocker_transition` broadcast here,
    // after both broadcasts above.
    return NextResponse.json({ task: mapTaskRow(fresh) })
  }

  // If neither branch matched, fall through to the generic write path below.
  // Example: PUT { status: 'awaiting_owner' } on an already-awaiting task,
  // or PUT { status: 'assigned' } as part of a non-blocker transition
  // (e.g., scheduler requeue). COMPAT-03: generic path keeps working.
}
```

**Key rules:**

1. **Insert the branch AFTER gate check + Aegis check** (lines 418-434) but BEFORE any `fieldsToUpdate.push('status = ?')` work (line 435+). Return early from the branches that fired; fall through ONLY when neither pause nor resume conditions matched.

2. **Short-circuit ownership:** the pause and resume branches OWN the status write + the blocker-column write entirely. They do NOT call into the generic write path. This is intentional — the generic path is a dynamic column builder and mixing the atomic envelope with other fields is a misuse. If the caller sent other fields alongside the blocker pause (e.g., `description`), they are IGNORED in this branch. This is an acceptable v1.3 limitation; the CONTEXT.md-deferred-items explicitly allows it.

3. **Recipe path rejection** — the `isRecipe` check fires FIRST on the pause branch. Recipe-tagged tasks have their own flow at `src/app/api/tasks/[id]/checkpoints/route.ts` (POST a `status: 'blocked'` checkpoint). A PUT that targets `awaiting_owner` on a recipe task is a caller mistake and returns 409 with a clear redirect message.

4. **Resume branch does NOT fire for recipe tasks either.** Recipe tasks that are `awaiting_owner` resume via owner PUT today (flipping to `assigned` triggers the runner to re-claim). The `!isRecipe` guard means recipe resumes still fall through to the generic write path — unchanged behavior. **COMPAT lock.**

5. **Concurrent-transition 409:** the WHERE-guarded UPDATE protects against a racing `in_progress → failed` (scheduler retry) or `awaiting_owner → assigned` (another PUT). `changes === 0` → the transaction throws and the handler returns 409.

6. **Assigned_to preserved through pause:** note the pause UPDATE does NOT touch `assigned_to`. On resume, the pre-blocker assignee is intact and the task re-enters `dispatchAssignedTasks` on the next scheduler tick.

7. **Event emission on commit:** the `task.status_changed` broadcast fires AFTER `run()` returns (outside the transaction closure). Plan 20-03 will append a second `task.blocker_transition` broadcast right after this one — leave a comment marker so that plan knows exactly where to land.

8. **The existing retry/fail paths are not touched.** `requeueStaleTasks` continues to flip stale `in_progress` tasks to `assigned` or `failed` without going through this PUT branch (it writes directly via `db.prepare(...).run(...)`). `dispatchAssignedTasks` catch-branch still writes `status='failed'` or `status='assigned'` on caught exceptions without touching the blocker envelope. Those code paths set `error_message` and `dispatch_attempts` but NOT `runner_last_failure_reason` (legacy path), so there is no cross-interference. **COMPAT-03 proof.**

9. **Fetch fresh row + mapTaskRow:** reuse the file's existing helpers. If the rest of the file uses `mapTaskRow(currentTask)` with merged fields, re-read the row from DB instead (simpler — less chance of desync with the blocker write).

Adding any new event type in this plan (`task.blocker_transition` is Plan 20-03's concern). Touching the recipe-path checkpoint handler (separate plan if at all).
