<objective>
Extend Phase 14's `seedMcDir` (at `src/lib/runner-worktree.ts:126-152`) to optionally append a single visible marker line to `<worktree>/.mc/progress.md` when a task resumes AFTER a blocker. Per 15-CONTEXT.md § "Blocker & Resume Flow", the marker line is:

```
<iso-ts> | <<< RESUMED AFTER BLOCKER: <blocker_reason> >>>
```

This lets the in-container agent reading progress.md naturally see the blocker reason without expanding the runtime env surface (no new env vars).

**Critical Phase 14 regression-free requirements:**
- First-attempt (is_resuming=false) seeding MUST remain unchanged — progress.md is written with just the header, no marker.
- Resume path (is_resuming=true) MUST STILL preserve existing progress.md and checkpoints.jsonl — the marker is APPENDED, never overwrites.
- Defensive fallback (is_resuming=true but no progress.md on disk — operator wipe scenario) MUST STILL create an empty placeholder file — with OR without the marker.
- `.gitignore` handling is unchanged (always rewritten to `"*\n"`).

**Contract change:** `seedMcDir` input type gains an optional `resume_marker?: { blocker_reason: string; at_iso: string } | null` field. `null` and `undefined` both mean "no marker". A present resume_marker on a first-attempt (is_resuming=false) call is a no-op — first attempts are never marker-prefixed.

Purpose: Wave 2 Plan 15-05 will build the blocker flow that sets `resume_marker` from the dispatch payload. This plan delivers the filesystem-side contract so Plan 15-05 can focus on the blocker state machine without touching runner-worktree.

Output: `src/lib/runner-worktree.ts` extended with marker append; dedicated test file covering all 6 state combinations.
</objective>
