# Phase 10 Spec — Multi-GSD Per Project (Native)

phase: 10-multi-gsd-per-project
status: verified
owner: hermes-ceo

## Goal
Enable one Mission Control project to run multiple concurrent GSD lifecycles (milestones/workstreams/phases/plans) natively, with independent state machines and shared roll-up visibility.

## Why
Single `project.gsd_phase` is structurally insufficient for real operations. A project needs continuous milestone growth, concurrent tracks, and parallel plan execution without forcing project fragmentation.

## Required Outcomes
1. Remove single-lifecycle bottleneck from project model.
2. Add first-class entities for milestone/phase/plan/workstream.
3. Preserve backward compatibility for existing Phase 9 data.
4. Support parallel lifecycles safely (locking, idempotency, conflict checks).
5. Keep existing role/gate semantics (operator/admin mutation, viewer read).

## Data Model (Target)

### Existing entities retained
- projects
- tasks

### New entities
1) `gsd_workstreams`
- id (pk)
- project_id (fk projects.id)
- key (text, unique per project)
- name (text)
- status (active|paused|complete)
- created_at, updated_at

2) `gsd_milestones`
- id (pk)
- project_id (fk)
- workstream_id (nullable fk)
- version_label (e.g. v1.2)
- title
- status (planned|active|complete|archived)
- started_at, completed_at
- created_at, updated_at

3) `gsd_phases`
- id (pk)
- milestone_id (fk)
- phase_key (text, e.g. 10, 10.1)
- phase_slug
- lifecycle_phase (discuss|plan|execute|verify|done)  # state machine pointer for THIS phase node
- ordering_numeric (real)  # supports decimal insertion
- status (planned|active|complete|deferred)
- depends_on_phase_ids (json)
- created_at, updated_at

4) `gsd_plans`
- id (pk)
- phase_id (fk)
- plan_ref (e.g. 10-01)
- title
- wave (integer)
- status (todo|in_progress|review|done|failed)
- depends_on_plan_ids (json)
- created_at, updated_at

### Task linkage extensions
Add nullable foreign keys on tasks:
- gsd_workstream_id
- gsd_milestone_id
- gsd_phase_id
- gsd_plan_id

Retain existing task fields for compatibility:
- gsd_phase (legacy enum field)
- gate_required / gate_status / gate_approved_by / gate_approved_at

## Compatibility / Migration Strategy

Phase A (additive):
- Create new tables + indexes + nullable FK columns.
- Keep existing project-level GSD columns untouched.

Phase B (dual-write):
- New endpoints write to new tables.
- Existing phase-9 endpoints continue to work by mapping to a default implicit milestone/workstream.

Phase C (read switch):
- Lifecycle UI reads new hierarchy first; falls back to legacy fields.

Phase D (optional deprecation):
- Mark project-level `gsd_phase` as legacy read-through; keep indefinitely unless explicit removal approved.

## Execution Semantics

1) Each phase node has its own D->P->E->V->Done machine.
2) Multiple phase nodes can be active in one project if they belong to different milestones/workstreams.
3) Same-workstream phase ordering enforced by `ordering_numeric` + dependency checks.
4) Plan wave parallelization allowed iff no conflict and dependency violations.
5) Gate-required tasks still block forward status transitions unless approved.

## Conflict and Safety Rules

- Reject same-wave plan execution when plans mutate conflicting resources (initially file-path hints in metadata; later semantic detector).
- Idempotency keys on create endpoints (`project_id+workstream_key+milestone+phase_key+plan_ref`).
- Transactional transitions with optimistic locking (`updated_at` precondition).

## Event Bus Extensions
Add events:
- gsd.workstream.created/updated/completed
- gsd.milestone.created/updated/completed
- gsd.phase.created/transitioned
- gsd.plan.created/transitioned
- gsd.conflict.detected

## UI Target

Project -> Lifecycle tab becomes hierarchical:
- Left: Workstreams + Milestones tree
- Center: Phase board (ordered + decimal inserts)
- Right: Plans-by-wave + gate queue
- Rollup badges at project top: active milestones, blocked gates, wave conflicts

## Non-goals (Phase 10)
- Full AI auto-roadmapper generation in MC UI
- Cross-project dependency graphing
- Automatic code-diff conflict detection (metadata-based in Phase 10)

## Acceptance Criteria
1. One project can host at least 2 active milestones concurrently.
2. Each milestone can have independent current phases and plan waves.
3. Parallel plan execution within a project works with dependency checks.
4. Legacy phase-9 projects still function with no data loss.
5. E2E proves: create workstream->milestone->phase->plans, transition independently, observe rollup updates.
