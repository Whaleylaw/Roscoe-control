# Phase 6 — UI Design Contract

> Visual and interaction contract for the Settings view inside the project workspace. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

Phase 6 replaces the 16-line `settings-view.tsx` stub with a three-section grouped form that edits the seven roadmap-scoped project fields (name, description, status, color, ticket_prefix, deadline, github_repo) and persists through `PATCH /api/projects/[id]`. No new component library, no new design tokens — the spec inherits the Mission Control dark-first void palette, Tailwind scale, and typography from Phase 1–5 and commits the form-control patterns already established in `project-manager-modal.tsx` (the UX reference). All text flows through the `project.settings.*` i18n namespace, translated atomically across 10 locales per the Phase 5 playbook.

## Design System

| Property | Value |
|----------|-------|
| Tool | none (project-defined system via Tailwind + CSS variables, inherited from Phase 5) |
| Preset | not applicable |
| Component library | none — hand-rolled primitives in `src/components/ui/` (button, loader, theme-background) on top of Tailwind 3.4.x |
| Icon library | **none** — CLAUDE.md forbids icon libraries. Settings copy is label-only; no glyphs on inputs. Unicode allowed for inline affordances only: `×` cancel, `•` bullet in helper text, `⚠️` reserved for error banner glyph at `text-destructive` |
| Font | `var(--font-sans)` (system UI stack); `var(--font-mono)` for ticket-prefix input only (matches Phase 5 ticket-ref monospace) |

**Rationale for Tool: none** — CLAUDE.md project rules override the gsd-ui default. Four prior phases (1–5) have shipped views using the established token system. Phase 6 introduces zero new tokens.

## Color

Inherited from Phase 5 void palette. No new tokens.

| Role | Token | Dark value (HSL) | Usage in this phase |
|------|-------|------------------|---------------------|
| Dominant (60%) | `bg-background` + `bg-surface-1` | `215 27% 4%` / `222 35% 7%` | Page background; input field background (`bg-surface-1`) |
| Secondary (30%) | `bg-card` / `bg-surface-2` + `border-border` | `220 30% 8%` / `220 30% 10%` + `220 20% 14%` | Section container backgrounds (if section cards used), sticky footer `bg-card/95 backdrop-blur`, all input borders |
| Accent (10%) | `text-primary` / `bg-primary` (void-cyan `187 82% 53%`) | — | **Reserved-for list below** |
| Destructive | `text-destructive` / `bg-destructive/10` / `border-destructive/30` | `0 72% 51%` | Inline field errors (border + text), top-of-form error banner, dirty-state "Unsaved changes" dot is NOT destructive (see accent list) |

**Accent (void-cyan) is reserved for — and ONLY for:**

1. **Primary Save button fill** — `bg-primary text-primary-foreground hover:bg-primary/90` (Button default variant) in the sticky footer only
2. **Selected color swatch ring** — `border-foreground scale-110` on the actively-selected swatch (matches `project-manager-modal.tsx:391`; not a primary color use but completes swatch affordance). Accent tint NOT applied to the swatch itself (those are palette hex values)
3. **Focus ring on any input** — `focus-visible:ring-2 focus-visible:ring-ring` (ring token maps to primary in dark theme)
4. **"Unsaved changes" indicator dot** — 8px `bg-primary rounded-full` preceding the footer's "Unsaved changes" label, signaling pending state without alarming the user

Accent is **not** used for:
- Section headings (those are `text-foreground`)
- Field labels (those are `text-foreground`)
- Helper text (that is `text-muted-foreground`)
- Select dropdown chevron, date-input calendar icon (UA default)
- Any border that is not a focus ring
- The Cancel button (secondary variant)

**Dirty-state visual language**

| Element | Idle | Dirty | Saving |
|---------|------|-------|--------|
| Field outline | `border-border` | unchanged (no per-field dirty chrome) | unchanged |
| Sticky footer | hidden | `sticky bottom-0 bg-card/95 backdrop-blur border-t border-border` visible | same, inputs disabled |
| "Unsaved changes" label | n/a | `text-xs text-muted-foreground` with preceding 8px `bg-primary rounded-full` dot | replaced with `Saving…` text, no dot |
| Save button | n/a | enabled `variant="default"` | `disabled` with inline text `Saving…` (no spinner — reuse `Loader` component only if it proves necessary during execution; default is text swap) |
| Cancel button | n/a | enabled `variant="secondary"` | `disabled` |

**Read-only mode (viewer role)**

| Element | Treatment |
|---------|-----------|
| Top-of-form note | `text-xs text-muted-foreground bg-surface-1 border border-border rounded-md px-3 py-2` at the very top of the form, above section 1 |
| All inputs | native `disabled` attribute — browser renders with `opacity-50 cursor-not-allowed` (Tailwind `disabled:` variant already set on Button; apply same pattern to inputs via `disabled:opacity-50 disabled:cursor-not-allowed`) |
| Sticky footer | never rendered in read-only mode |
| Color swatches | `pointer-events-none opacity-60` |

## Layout Contract

### Root shell

```
<div className="p-6 max-w-3xl pb-12">           // view root — matches other workspace views' padding
  <h2 className="text-lg font-semibold mb-6">   // view title (reuses existing project.settings.title key)
  {readOnlyNote}                                 // conditional, viewer role only
  {errorBanner}                                  // conditional, top-of-form only
  <form className="space-y-6">                  // sections gap (24px)
    <section>Basics</section>
    <section>Appearance & Tracking</section>
    <section>Integrations</section>
  </form>
  {stickyFooter}                                // fixed to bottom of form when dirty
</div>
```

Max-width constraint: `max-w-3xl` (768px). Keeps line-length readable for textarea and prevents inputs from stretching across ultra-wide viewports. At narrower viewports the form naturally shrinks; at narrower than `md` (768px) the two-column grid inside sections collapses to one column (see Responsive).

### Section structure

Each section renders as a vertical block (no surrounding card border — keeps the form feeling like one continuous sheet, contrasting with the modal's list-row chrome):

```
<section className="space-y-4">
  <div className="space-y-1">                    // header cluster
    <h3 className="text-lg font-semibold">       // section heading
    <p className="text-xs text-muted-foreground"> // optional section hint (deferred if empty)
  </div>
  <div className="grid grid-cols-1 md:grid-cols-2 gap-4">  // field grid
    <FieldBlock />
    ...
  </div>
</section>
```

Sections are separated by 24px (`space-y-6` on the parent form). No horizontal rule between sections — the 24px gap + typographic weight difference between `text-lg font-semibold` heading and following fields is sufficient separation.

### FieldBlock anatomy

```
<div className="space-y-2">                       // label → input → helper/error
  <label className="block text-sm font-semibold text-foreground">
  <input|textarea|select
    className="w-full bg-surface-1 text-foreground border border-border rounded-md px-3 py-2 text-sm
               focus-visible:outline-none focus-visible:ring-2 focus-visible:ring-ring focus-visible:ring-offset-2
               focus-visible:ring-offset-background
               disabled:opacity-50 disabled:cursor-not-allowed
               aria-invalid:border-destructive aria-invalid:focus-visible:ring-destructive"
    aria-describedby={helperId}
    aria-invalid={hasError}
  />
  {helperText && <p id={helperId} className="text-xs text-muted-foreground">{copy}</p>}
  {errorText && <p className="text-xs text-destructive" role="alert">{copy}</p>}
</div>
```

Helper and error are mutually exclusive in the visual stack when both present: error replaces helper. Field still announces its describedby helper id if helper text exists; aria-errormessage is not used (the `role="alert"` on the error paragraph carries assistive-tech notification).

**Section 2 — Appearance & Tracking**

| Field | Control | Grid | Notes |
|-------|---------|------|-------|

"None" pill is a text button with `bg-transparent border border-border rounded-full px-3 py-1 text-xs` that selects the null-color state |
| Ticket prefix | `<input type="text" className="font-mono uppercase">` | `md:col-span-1` | Helper text: *"Changing the prefix affects only new tickets. Visually uppercases the input via CSS `uppercase` (form value is sent raw; server normalizes) |

### Sticky footer

Only rendered when `isDirty === true` AND `role !== 'viewer'`.

```
<div className="sticky bottom-0 -mx-6 mt-6 px-6 py-4 bg-card/95 backdrop-blur border-t border-border
                flex items-center justify-between gap-4">
  <div className="flex items-center gap-2 text-xs text-muted-foreground">
    {isSaving
      ? <span>Saving…</span>
      : <><span className="inline-block w-2 h-2 bg-primary rounded-full" aria-hidden="true" />
          <span>{t('settings.unsavedChanges')}</span></>}
  </div>
  <div className="flex items-center gap-2">
    <Button variant="secondary" size="md" onClick={cancel} disabled={isSaving}>
      {t('settings.cancel')}
    </Button>
    <Button variant="default" size="md" onClick={save} disabled={isSaving || !canSave}>
      {isSaving ? t('settings.saving') : t('settings.save')}
    </Button>
  </div>
</div>
```

`-mx-6` negates the parent `p-6` so the footer's top border spans the full view width. The footer sits `sticky bottom-0` inside the view's scroll container, not `fixed` — this keeps it scoped to the Settings tab without interfering with the workspace shell's breadcrumb/tab bar.

### Dirty detection

Dirty = any field differs from the last-loaded project value after both sides are normalized:

- Strings: trim
- Ticket prefix: uppercase + strip non-alphanumeric + 12-char cap (mirror `normalizePrefix` from route.ts:11-14) before comparison — prevents false-dirty after save when server echoes normalized value
- Deadline: compare the date string representation (YYYY-MM-DD) against the stored Unix-seconds converted to the same string; empty string === null
- Color: empty string === null; hex strings compared literally
- GitHub repo: trim; empty string === null

Save is disabled when `!isDirty || trimmedName.length === 0 || isSaving`.

### Focus management

- On mount: no autofocus (respects scroll position, avoids jumpiness). First field (Name) is keyboard-first via natural tab order.
- On save success: focus remains where the user left it (active input or Save button). Name field is NOT auto-focused. The footer swaps from "Saving…" to hidden; if Save button had focus, focus falls back to the section that contained the last-edited field via `document.body` as the default (browser behavior is acceptable; no manual `ref.focus()` required).
- On save error (banner shown): focus moves to the error banner (`tabIndex={-1}` + `ref.focus()`) so screen readers announce it. Banner is `role="alert"`.

## Interaction Contract

### Color picker

Exact behavior matches `project-manager-modal.tsx:386-396`:

| Action | Result |
|--------|--------|
| Click unselected swatch | Sets `color` to swatch hex; adds `border-foreground scale-110` |
| Click currently-selected swatch | Clears `color` to empty string (serialized to null on save); reverts to `border-transparent` |
| Click "None" pill | Explicitly clears `color` to empty string |
| Keyboard: Tab through swatches | Each swatch is a `<button type="button">` in natural tab order; focus ring `focus-visible:ring-2 focus-visible:ring-ring focus-visible:ring-offset-2` |

No 9th swatch — palette is exactly the 8 hex strings in `COLOR_PALETTE` duplicated into `settings-view.tsx` (per D-11).

### Status select

Native `<select>`. Options:

```
<option value="active">{t('settings.statusActive')}</option>
<option value="archived" disabled={slug === 'general'}>{t('settings.statusArchived')}</option>
```

Native-disabled option remains keyboard-navigable but not selectable. No custom dropdown — native consistency across browsers and screen readers outweighs visual polish for v1.

### Save flow

1. User clicks Save (or presses Enter inside a single-line input, which triggers form submit).
2. Form enters `isSaving = true`; all inputs get `disabled`; footer shows `Saving…`.
3. Build PATCH body by including ONLY fields that differ from loaded values, plus `name` always (server requires it only if it changed, but sending the current name is harmless and simplifies). Empty strings for description/github_repo/color sent as explicit `""` (server coerces to null per route.ts:133, 154, 162).
4. `fetch('/api/projects/' + projectId, { method: 'PATCH', ... })`
5. On 2xx: read `project` from response body; refresh `/api/projects` → update Zustand `projects[]`; clear dirty state; leave form values at server-echoed values; footer hides.
6. On 400/409: parse JSON `error` string; route to inline field error (see Error mapping below).
7. On other non-2xx / network error: show top-of-form banner with server error text or fallback *"Failed to update project"*.

### Cancel flow

Immediate: reset all form state to the last-loaded project values; clear dirty state; clear any top-of-form banner; footer hides. No confirmation prompt (no data loss — nothing was saved).

### Real-time updates

Not wired for Phase 6 (D-19). If the project object changes in Zustand via another mechanism (e.g., the project-manager-modal from the global nav), the settings-view should reconcile:

- If the form is pristine: re-sync all fields to new values.
- If the form is dirty: keep user's in-progress edits. Do NOT clobber typed-in values. No banner shown (out of scope for v1 — silent reconciliation favors the editing user).

### Success state

**No toast.** Per D-05 / D-17, on success the footer hides, the form stays populated with server-echoed values, and the Zustand store refresh propagates to the breadcrumb/dashboard. Silent success is the contract — matches the sessions/tasks scoped views which also silently reconcile via SSE or store refresh without toast chrome.

### Destructive actions

**There are no destructive actions in Phase 6.** Delete, archive-default-project, and unassignment are all out of scope. The Status select's Archived option is disabled for the general project (preventing the only edit that would have destructive semantics); no confirmation dialogs are rendered by this view.

## Empty / Null / Loading States

| State | Treatment |
|-------|-----------|
| Initial project fetch loading (`project == null`, `loading === true`) | Render view root with a single centered `<Loader />` component (existing `src/components/ui/loader.tsx`). No skeleton; matches the dashboard/tasks/sessions loading pattern — a loader primitive is sufficient for sub-second fetches |
| Project fetch error | Inline error card at the top of the view replacing the form entirely: heading `Could not load project settings` (i18n key `project.settings.loadErrorHeading`), body `Retry` button that calls the context's reload. **This path reuses** existing `project-context.tsx` error state — the view does not own fetching |
| Null deadline on load | Date input renders empty (`value=""`); saving an empty value sends `deadline: null` |
| Null github_repo on load | Text input renders empty; saving sends `github_repo: null` |
| Null color on load | No swatch has selected styling; "None" pill gets `aria-pressed="true"` |
| Empty description on load | Textarea renders empty; saving sends `description: null` via trim-to-empty-to-null logic in the PATCH body |

Add two i18n keys for the load-error inline card:

```
project.settings.loadErrorHeading   = "Could not load project settings"
project.settings.loadErrorRetry     = "Retry"   (reuse project.common.retry if it exists — Phase 5 introduced this key)
```

Prefer reusing `project.common.retry` from Phase 5. If the key is already present in `messages/en.json`, do NOT add `project.settings.loadErrorRetry` — reference `project.common.retry`.
