**Design invariant:** All four components are PURE controlled inputs. They accept `value` + `onChange` props and NEVER mutate parent state, NEVER own fetch state beyond the autocomplete results, and NEVER bypass next-intl. The parent CreateTaskModal / EditTaskModal owns the full form state.

**Step 1 — `src/components/panels/task-form/recipe-combobox.tsx`:**

Extend the `MentionTextarea` pattern (task-board-panel.tsx:199-339 — keyboard handling / open state / absolute-positioned dropdown). Key adaptations:
- Data source is `GET /api/recipes/search` with 300ms debounce (autocomplete results).
- **Name lookup for the currently-selected slug comes from the Zustand `recipes` slice created by Plan 16-02 (Task 1).** RecipeCombobox subscribes via `useMissionControl(s => s.recipes)` and looks up the selected row by slug to render the name inside the input. Falls back to the slug literal if the cache is not yet hydrated (identical fallback pattern to RecipeBadge in 16-02).
- `AbortController` per fetch to cancel stale in-flight requests (Pitfall 5 — rapid-typing race).
- Single-select with `Clear` button to null-out.
- Each autocomplete result row shows recipe name + model-tier chip (reuse `MODEL_TIER_COLORS` via `modelTierClassName(modelToTier(recipe.model?.primary))`) + truncated description.

**Integration contract with 16-02 (LOCKED):**
- RecipeCombobox READS the `recipes` slice from `useMissionControl` — it does NOT fetch `/api/recipes`, does NOT call `refreshRecipes()`, and does NOT subscribe to `mc:recipe-*` DOM events. That boot + refresh lifecycle is owned exclusively by Plan 16-02 (Task 1) inside `useServerEvents`. Duplicating it here would double-fetch on boot.
- Only the `/api/recipes/search` autocomplete fetch is owned by this component.
- If `useMissionControl(s => s.recipes)` returns `[]` at first paint (pre-hydration), the input displays the raw slug — once 16-02's boot refresh populates, the next render picks up the friendly name. Document this in tests.

Props (LOCKED — no `valueName` prop; the name comes from the shared Zustand slice):
```typescript
type RecipeComboboxProps = {
  value: string | null                              // current selected slug, null = no recipe
  onChange: (slug: string | null) => void
  disabled?: boolean                                // RECIPE_LOCKED gate
  lockedHint?: string                               // localized hint when disabled
  label?: string                                    // field label (falls back to t('label'))
}
```

Implementation essentials:
- Input ref for focus management.
- State: `{ q, open, activeIndex, results, fetching }`. No `selectedRecipe` state — derived from `value` + Zustand `recipes`.
- `useEffect` runs 300ms debounced fetch when `q` changes and `open` is true.
- AbortController cancels on unmount and query-change.
- Render: `<div role="combobox" aria-expanded={open} aria-controls={listId}>` + `<input>` + absolute dropdown with `role="listbox"` and items with `role="option" aria-selected`.
- Keyboard handling mirrors MentionTextarea: ↑/↓ cycle modular, Enter selects, Escape closes, Tab closes without select.
- When selected (value != null), input shows `recipes.find(r => r.slug === value)?.name ?? value` (i.e. friendly name with slug fallback) + a `✖` clear button that calls `onChange(null)`.
- When disabled, input is readonly and the clear button hidden; render `{lockedHint}` below the field (red/amber muted).
- i18n: `useTranslations('taskBoard.recipeField')` → keys: `label`, `placeholder`, `clear`, `noResults`, `lockedHint`, `searching`.

Accessibility (research Focus Area 13):
- `aria-activedescendant` pointing to the active option's id.
- `aria-label` on the clear ✖ button.
- `role="combobox" aria-autocomplete="list"` on the input.
