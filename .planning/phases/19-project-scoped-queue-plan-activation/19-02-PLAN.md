**Transition route WIP (src/app/api/gsd/plans/[plan_id]/transition/route.ts, +59 lines):**

Points where the WIP diverges from CONTEXT.md locked decisions (MUST be corrected):

1. **Wrong source set.** WIP filters `status IN ('backlog', 'awaiting_owner')`. CONTEXT specifies strict source set `('backlog', 'todo')` and explicitly states `awaiting_owner` is NEVER auto-activated by plan transition.

2. **No assignee-based routing.** WIP always sets `status = 'inbox'`. CONTEXT specifies: `assigned_agent_id` non-null OR `recipe_slug` non-null → route to `'assigned'`; otherwise → `'inbox'`. The recipe-slug branch is critical for v1.2 runtime parity.

3. **No dead-assignee policy.** CONTEXT specifies: if `assigned_agent_id` points at an agent whose status indicates it's unavailable/terminal (the codebase does NOT currently have a `disabled` status — agents use `offline/idle/busy/error`; Task 1 must investigate and pick the right predicate, or degrade gracefully if no such status exists), clear `assigned_agent_id`, route to `inbox`, and count as `reassigned`. Research during execution: either treat `error` as the terminal state, or document this as a no-op in v1.3 and surface the limitation in the summary.

4. **Missing counters in payload.** WIP returns `{activated_count, task_ids}`. CONTEXT specifies full shape `{activated, already_active, skipped_by_state, reassigned, by_status: {inbox, assigned}, task_ids}`. The keys `activated` (not `activated_count`) and the four counter buckets are CONTRACTUAL — Phase 23's acceptance test will assert this shape.

5. **Wrong event name.** WIP emits `gsd.plan.queue_activated`. CONTEXT specifies `gsd.plan.tasks_activated`. Also, WIP emits a per-task `task.status_changed` broadcast inside the loop — this is acceptable but noisy; preserve it only if other parts of the code depend on per-task broadcasts (quick grep to verify), otherwise drop to keep the activation atomic from an observer's perspective.

6. **No transaction wrap.** WIP runs the plan UPDATE outside a transaction, then loops task UPDATEs individually. CONTEXT specifies: "One SQLite transaction wraps the plan status flip and all task activations. Real DB / validation errors roll back the entire transition." Use `db.transaction(() => { … })()` per better-sqlite3 idiom.

7. **No `skipped_by_state` / `already_active` counting.** WIP just loops activatable rows. Spec requires querying ALL plan-linked tasks and bucketing them:
   - source state in {backlog, todo} → activate, increment `activated`, break down by resulting status in `by_status`
   - current state in {inbox, assigned} → increment `already_active` (no UPDATE)
   - current state otherwise (review, in_progress, failed, awaiting_owner) → increment `skipped_by_state` (no UPDATE)
   - dead-assignee recovery that routed to inbox → also increment `reassigned`

8. **Missing EventType union entry.** `src/lib/event-bus.ts` declares a strict `EventType` literal union consumed by `eventBus.broadcast(type: EventType, …)`. It currently does NOT include `'gsd.plan.tasks_activated'`, so the renamed broadcast will fail `pnpm typecheck` until the union is extended. Task 1 must append the new literal to the union.

The WIP's existing structure (pre-write gate checks, optimistic lock, canTransitionGsdPlanStatus) is CORRECT and MUST be preserved. Only the `if (toStatus === 'in_progress') { … activation block … }` and its response-body assembly need rewriting.

AND status IN ('backlog','todo')` (the AND clause makes the UPDATE a no-op if status drifted mid-transaction — which inside a transaction shouldn't happen, but the belt-and-suspenders WHERE keeps the operation safe).

- `row.status IN ('inbox', 'assigned')` → increment `already_active`, no UPDATE.
- `row.status` in any other state (review, in_progress, awaiting_owner, failed, done, cancelled, etc.) → increment `skipped_by_state`, no UPDATE.
