<objective>
Add the `runner` auth principal — backed by an auto-generated `.data/runner.secret` file — and wire it into `getUserFromRequest` with a hard path-prefix check. A request presenting this secret gets operator-level access to `/api/runner/*` endpoints and NOTHING else. This is Wave 1 substrate for RAUTH-01.

Purpose: The runner daemon (Phase 14) is a separate Node process that talks to MC over HTTP. It needs a long-lived identity. This plan stands up that identity without introducing any runner behavior — we ship the lock but nothing opens it yet.

Output:
- `.data/runner.secret` auto-generates on first boot with 0600 perms and ≥32 bytes of entropy
- `src/lib/runner-secret.ts` — small focused module (read, generate, persist)
- `src/lib/auth.ts` — extended `getUserFromRequest` with a new branch BEFORE the existing API-key branch that (a) only matches when `url.pathname.startsWith('/api/runner/')`, (b) constant-time compares the bearer to the runner secret, (c) returns a synthetic `runner` user
- Safe audit logging — structure matches existing `global_api_key_used` pattern
</objective>

Requirements:
1. Use `randomBytes(32)` and `.toString('base64url')` for the wire format (44 chars, URL-safe). Encoding choice is Claude's discretion per CONTEXT.md — base64url works cleanly in `Authorization: Bearer <value>`.
2. Write with `{ mode: 0o600 }` via `fs.writeFileSync`. On POSIX this sets permissions at create time; on an existing file it will NOT narrow permissions, so the sequence for a fresh generate is: `fs.rmSync(path, { force: true })` then `fs.writeFileSync(path, secret, { mode: 0o600 })`. Add a comment explaining why.
3. Ensure directory via `ensureDirExists(config.dataDir)` (already exported from config.ts — reused by auto-credentials).
4. Skip generation when `process.env.NEXT_PHASE === 'phase-production-build'` (match auto-credentials.ts pattern — avoids writing during `next build`).
5. `getRunnerSecret()`: `readFileSync(path, 'utf8').trim()` → if empty or decoded-length < 32 return `null`. Do NOT auto-generate from `getRunnerSecret`; that is `ensureRunnerSecret`'s job. Separation of read vs. provision matters for test isolation.
6. Log line on generation: `logger.info('Auto-generated runner secret (persisted to .data/runner.secret — mode 0600)')`. On re-read success: NO log (keep boot logs quiet).
7. Export exactly: `ensureRunnerSecret`, `getRunnerSecret`, `RUNNER_SECRET_FILENAME`.

2. Add a new branch INSIDE `getUserFromRequest(request: Request)`, placed AFTER the proxy-auth block but BEFORE the session-cookie block. It must be the first auth mechanism checked for /api/runner/* paths — so path-prefix gating gets first say:

```typescript
// Runner principal — auto-generated .data/runner.secret, strictly scoped to /api/runner/*
// See RAUTH-01 + .planning/phases/11-runtime-foundation-v1-2/11-CONTEXT.md
const url = new URL(request.url)
if (url.pathname.startsWith('/api/runner/')) {
  const bearer = extractApiKeyFromHeaders(request.headers)  // reuse existing helper
  const runnerSecret = getRunnerSecret()
  if (bearer && runnerSecret && safeCompare(bearer, runnerSecret)) {
    try {
      logSecurityEvent({
        event_type: 'runner_auth',
        severity: 'info',
        source: 'auth',
        // DO NOT include the secret value, a prefix, or a hash of it. CONTEXT.md locks this.
        detail: JSON.stringify({ principal: 'runner', path: url.pathname, method: request.method }),
        ip_address: request.headers.get('x-real-ip') || 'unknown',
        workspace_id: getDefaultWorkspaceContext().workspaceId,
        tenant_id: getDefaultWorkspaceContext().tenantId,
      })
    } catch { /* startup race / DB not ready — non-fatal */ }
    return {
      id: -1000,  // negative sentinel outside the 1..N user range AND distinct from agent-scoped keys (which are -agent_id); -1000 is out of range for realistic agent_id
      username: 'runner',
      display_name: 'Runner Daemon',
      role: 'operator',  // operator level — runner needs write access to checkpoints/claim endpoints (Phase 14/15) but is not admin
      workspace_id: getDefaultWorkspaceContext().workspaceId,
      tenant_id: getDefaultWorkspaceContext().tenantId,
      provider: 'local',
      email: null,
      avatar_url: null,
      is_approved: 1,
      created_at: 0,
      updated_at: 0,
      last_login_at: null,
      agent_name: null,
    }
  }
  // Fall through. If the path is /api/runner/* but the bearer is wrong or absent,
  // we intentionally do NOT short-circuit — a request could carry a runner-token
  // (Plan 11-04) or a valid session cookie. The subsequent branches handle those.
}

// IMPORTANT: DO NOT allow the runner-secret to succeed on paths OUTSIDE /api/runner/*.
// The url.pathname.startsWith gate above is the ONLY check — if it fails, we never
// compare the bearer against the runner secret at all. That means a request hitting
// /api/tasks with the runner secret presented as a bearer will fall through to the
// session-cookie / agent-API-key branches and be rejected there.
```

3. Do NOT modify the existing API-key admin path, the session-cookie path, the agent-scoped API-key path, the proxy-auth path, or `requireRole()`. Minimal diff — one new branch, two imports.

4. Do NOT add the runner-token principal here — that is Plan 11-04 and depends on the `task_runner_tokens` migration (Plan 11-03).
