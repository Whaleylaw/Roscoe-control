### Pattern 2: Env Var / Exit Code / Endpoint Reference Table

**What:** Every runtime surface exposes a stable name → value → source mapping. Use a 4-column table: **Name | Type | Default | Source-of-truth**. The fourth column is a relative link to the file + line where the name is defined.
**When to use:** Env vars, exit codes, HTTP endpoints, error codes, setting keys. In `docs/cli-agent-control.md` the "Exit code contract" is the precedent (3-column there; extend to 4 with source-of-truth here).
**Example:**

```markdown
## Runner daemon environment variables

| `MC_URL` | URL | `http://127.0.0.1:3000` | [`scripts/mc-runner.mjs:49`](../../scripts/mc-runner.mjs#L49) |
```

**What:** Every operational claim ("the runner authors `/recipe/PREAMBLE.md`") pairs a short prose sentence with a fenced code block pulled verbatim or near-verbatim from the source-of-truth file. **When to use:** Every behavioral claim. Especially important for the agent contract (preamble structure, checkpoints.jsonl format) where a paraphrase could mislead an agent author.
**Example:**

### Pattern 4: "Operator journey" end-to-end narrative for DOC-TUT

**What:** The `getting-started.md` doc walks through the full operator journey exactly once, end-to-end, with every command a reader can copy-paste into a terminal. Not a reference. Not a deep dive. Follow `docs/quickstart.md` structure (numbered steps, code fences, expected output blocks).
**When to use:** DOC-TUT only. Every other doc is a reference.
**Shape:**
1. Install + boot Mission Control
2. Configure `runtime.mount_allowlist` + `runtime.project_repo_map` via `pnpm mc settings set`
3. Author a minimal recipe (copy `recipes/hello-world/` structure)
4. Build `mc-hello-world-agent:latest` (`pnpm mc:build-hello-world`)
5. Start the runner (`node scripts/mc-runner.mjs`)
6. Create a task with `recipe_slug: hello-world` via curl or UI
7. Watch Progress tab update; see checkpoints arrive; see `review` → `done` flip
8. Inspect `.data/runner/logs/task-<id>/latest/stderr.log`

### Anti-Patterns to Avoid

- **Duplicating authoritative content.** `scripts/README.runner.md` already documents the runner daemon end-to-end (154 lines) at high quality. Do NOT rewrite it into `docs/runtime/runner-daemon.md`. Instead, `docs/runtime/runner-daemon.md` is a thin operator-facing intro (what it is, why you need it, where to find the deep reference) that LINKS to `scripts/README.runner.md` for the 90% of its content.
- **Quoting the design spec.** `docs/superpowers/specs/2026-04-18-recipe-agent-system-design.md` is design-era and contains claims that no longer match shipped code. Do NOT cite it as authoritative. Only quote from actually-shipped source files (under `src/`, `scripts/`, `docker/`, `recipes/`).
- **Inline "see Phase 14 Plan 14-08b" references.** The `.planning/` directory is internal. Operator docs must not depend on its existence or structure. Cite source files, not planning artifacts.
- **Stale examples.** Every recipe.yaml snippet, curl command, and env var reference MUST compile/parse against the shipped code. The harness enforces this.
- **API endpoints without method + path template.** Always `POST /api/runner/tasks/:task_id/submit`, never "the submit endpoint."
- **Screenshots without alt text.** Accessibility floor matches existing `docs/` — every `![img](path)` has human-readable alt.
- **Tool-specific agent assumptions.** The runtime is tool-agnostic (see user memory file: "Agent runtime is tool-agnostic"). Docs must NOT assume Claude Code, Aegis, or any specific agent SDK. The hello-world reference explicitly avoids it (no model calls, pure substrate verification).

## Common Pitfalls

### Pitfall 1: Submit→Done vs Submit→Review (Phase 17-01 drift)

**What goes wrong:** Operator reads a doc saying "agent POSTs submit and task flips to done" and the system observably goes to `review` instead. They think something is broken.
**Why it happens:** Phase 17 Plan 17-01 (RTEST-02, commit `e9e5fc1`) flipped the submit route from `SET status='done'` to `SET status='review'`. Phase 18-03 fixed this drift in seven Phase 14 markdown files (commit `b0b9c21`). The reference image `docker/hello-world-agent/agent.mjs:89` still POSTs `{status: 'done'}` (the agent's declaration of intent) and the route translates to review — the body literal is stable.
**How to avoid:** Every mention of the submit flow in `docs/runtime/agent-contract.md` and `docs/runtime/task-board-surfaces.md` MUST describe the **two-hop terminal transition**: `agent POST submit {status:'done'} → task flips in_progress → review → runAegisReviews() approves → review → done`. Cite `src/app/api/runner/tasks/[task_id]/submit/route.ts` (the flipper) AND `src/lib/task-dispatch.ts:414` (the Aegis reviewer) together. Never describe the agent's POST as "flipping to done."
**Warning signs:** Any prose in a draft doc that says "agent completes task" or "submit marks task done" without mentioning `review` is wrong.

### Pitfall 2: Runner-token allowlist has SEVEN entries not six (Phase 15 CP-01)

**What goes wrong:** Docs list the six original runner-token allowlist endpoints and miss that `POST /api/tasks/:id/checkpoints` was added in Phase 15 (CP-01).
**Why it happens:** The original `src/lib/runner-tokens.ts:8-9` comment said "DO NOT add entries" — now explicitly revoked for the CP-01 exception. Phase 15-01 extended the list from 6 to 7 entries (`e0e30e8`).
**How to avoid:** `docs/runtime/agent-contract.md` MUST enumerate all 7 entries. Reference the array at `src/lib/runner-tokens.ts:18-27` verbatim. The harness should grep-count the regex entries in that file and assert the doc table has the same count.
**Warning signs:** Any table listing runner-token-reachable endpoints with exactly 6 rows.

### Pitfall 3: `max_attempts` is NOT in the recipes DB row

**What goes wrong:** An operator reads a doc that says "recipe's max_attempts is stored in the recipes table" and tries to query or update it via SQL.
**Why it happens:** Phase 14 locked that `max_attempts` is intentionally NOT round-tripped through the DB row — the claim route re-reads `recipe.yaml` from disk every time. `getIndexedRecipeBySlug` projects a fixed column set; the `recipes` table has no `max_attempts` column.
**How to avoid:** `docs/runtime/recipes.md` schema section MUST call out that `max_attempts` is a filesystem-only field used at claim time. Cite `src/lib/runner-claim.ts:4-19` comment block.
**Warning signs:** Any reference to `SELECT max_attempts FROM recipes`.

### Pitfall 4: `runtime.project_repo_map` is the ONLY path to resolve project→repo

**What goes wrong:** Operators try to set a `PROJECT_REPO_MAP` env var on the runner, or expect the runner to auto-detect repos from `tasks.project_id`.
**Why it happens:** Phase 14-08b locked "No env-var fallback — a misconfigured `/api/runner/config` must fail loud at boot." `.planning/STATE.md:242` records this decision. The daemon resolves `workspace_source.project_id` to a local git repo path EXCLUSIVELY via the `runtime.project_repo_map` admin setting.
**How to avoid:** `docs/runtime/admin-config.md` must document this as the single-source-of-truth rule, and show the exact `pnpm mc settings set runtime.project_repo_map '{"1":"/abs/path"}'` command. Mention SIGHUP re-fetches the map without daemon restart.
**Warning signs:** Any reference to an env var like `RUNNER_PROJECT_REPO_MAP` or to reading the mapping from a file.

### Pitfall 5: Host networking — container uses `host.docker.internal`, not `localhost`

**What goes wrong:** Agent author copies a curl example with `http://localhost:3000` into their container code and it 404s or hangs.
**Why it happens:** The runner composes `MC_API_URL = http://host.docker.internal:<PORT>` and passes it to the container via `--add-host host.docker.internal:host-gateway`. Containers cannot reach the host's `localhost`. Phase 14-05 locked this.
**How to avoid:** `docs/runtime/agent-contract.md` MUST open the HTTP section with "Inside the container, `$MC_API_URL` is `http://host.docker.internal:<port>`, NOT `localhost`." Show the `--add-host` flag in the docker run example.
**Warning signs:** Any agent code example using `http://localhost` or `http://127.0.0.1` inside a container.

### Pitfall 6: Secrets never flow on docker argv (CONTAINER-01 invariant)

**What goes wrong:** An operator "simplifies" the docs by suggesting `docker run -e ANTHROPIC_API_KEY=$SECRET` and a reviewer approves it.
**Why it happens:** CONTAINER-01 invariant says no `--env` flag on the argv carries the `MC_API_TOKEN` value; secrets flow via `--env-file` only. Unit test `src/lib/__tests__/runner-docker.test.ts` asserts this by scanning argv.
**How to avoid:** The secrets section of `docs/runtime/runner-daemon.md` and `docs/runtime/admin-config.md` MUST explicitly say "never pass secrets on `docker run` argv; use `--env-file` or recipe-declared `secrets: [NAME]` list." Show the recommended pattern: `install -m 0600 /dev/stdin .data/runner/secrets/NAME <<<'value'`.
**Warning signs:** Any example using `docker run -e SECRET=value`.

### Pitfall 7: Standalone mode breaks recipe indexing by default

**What goes wrong:** Operator does `pnpm build && node .next/standalone/server.js` for production-parity, then wonders why their `recipes/` tree is empty.
**Why it happens:** `getRecipesRoot()` defaults to `<process.cwd()>/recipes`; standalone's cwd is `.next/standalone/` which does not contain the authored recipes tree. Phase 14-10 caught this.
**How to avoid:** `docs/runtime/admin-config.md` must document `MISSION_CONTROL_RECIPES_DIR` as required for standalone deployments, pointing at the repo's `recipes/` absolute path.
**Warning signs:** Any "deploy to production" section that doesn't mention `MISSION_CONTROL_RECIPES_DIR`.

### Pitfall 8: `recipe.secrets` is a list of ENV VAR NAMES, not values

**What goes wrong:** Operator reads a recipe.yaml example with `secrets: [ANTHROPIC_API_KEY]` and thinks `ANTHROPIC_API_KEY` is literally the string "ANTHROPIC_API_KEY" being passed, not a reference to a stored secret.
**Why it happens:** `src/lib/recipe-schema.ts:54` defines `secrets: z.array(z.string().min(1))` — an array of names. Runner resolves values from `.data/runner/secrets/<NAME>` at claim time.
**How to avoid:** Every recipe.yaml example in docs MUST annotate `secrets:` with a comment block: `# ENV VAR NAMES ONLY — runner resolves values from .data/runner/secrets/<NAME>`.
**Warning signs:** A recipe.yaml example showing `secrets: [ANTHROPIC_API_KEY=sk-xxx]` or similar value-in-recipe patterns.

### Pitfall 9: Runner-status UI lies when Docker is down

**What goes wrong:** UI shows 🟢 runner online but tasks never dispatch, because the runner exited with code 2 (Docker unreachable) and LaunchAgent is respawning every 30s — within the 90s stale window.
**Why it happens:** `runner_heartbeats` is UPSERTed on boot; a heartbeat persists even when `docker info` fails on the next attempt. The 90s stale window on the `last_heartbeat_at` column only registers "offline" after three failed restart cycles.
**How to avoid:** `docs/runtime/task-board-surfaces.md` MUST document the 90s stale window and note that "online" means "heartbeat fresh within 90s," not "Docker reachable right now." Cross-link to troubleshooting in `scripts/README.runner.md:135-140` (runner exits 2 → Docker daemon is not reachable).
**Warning signs:** Any prose saying "green banner means the runner is healthy" without qualification.

### Pitfall 10: `.mc/progress.md` and `.mc/checkpoints.jsonl` are append-only across attempts

**What goes wrong:** Agent author on attempt 2 reads `progress.md` thinking it's fresh, then writes fresh content on top without preserving attempt-1 history.
**Why it happens:** WORK-04/WORK-05 locked that progress.md and checkpoints.jsonl are preserved across attempts; seedMcDir preserves them on resume and always rewrites only task.json. Phase 17-05 byte-asserts append-only with `expect(jsonlAfterResume.slice(0, jsonlAfterKill.length)).toBe(jsonlAfterKill)`.
**How to avoid:** `docs/runtime/agent-contract.md` progress/checkpoint section MUST use the word "append" not "write." Show `fs.appendFileSync` in code examples, never `fs.writeFileSync` (except for HELLO.md-style brand-new files in the workspace).
**Warning signs:** Any code example using `writeFileSync(progressPath, ...)`.
