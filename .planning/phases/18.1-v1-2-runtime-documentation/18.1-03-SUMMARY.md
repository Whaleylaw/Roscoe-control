## Decisions Made

- **Submit lifecycle described as two-hop; body literal stable.** The plan's primary invariant — never describe submit as "marking the task done" — is enforced via a standalone ⚠️ callout at the top of the Submit section, a two-hop transition diagram, and a cite-both strategy (flipper AND reviewer) in every mention of review→done.
- **Aegis allowed as a server-side reference, not an SDK.** The plan's must-haves truth #8 forbids assuming "Claude Code, Aegis, or any specific agent SDK." I interpreted this as scoped to the agent-author's side: the recipe-image contract itself makes no Aegis assumption (the agent just POSTs `{"status":"done"}` and exits). Mentioning `runAegisReviews()` as the **server-side reviewer that performs the second hop** is factually correct and matches the plan's `key_links` entry which explicitly requires a citation to `src/lib/task-dispatch.ts (runAegisReviews)`. The doc never instructs the recipe author to depend on Aegis — only to know that review→done is driven server-side.
- **`/api/runner/checkpoint` preamble reference kept as a legacy alias.** The preamble generator at runner-preamble.ts:69-94 emits the older `/api/runner/checkpoint` URL for back-compat. Rather than rewrite the preamble text (which is source-of-truth), the doc quotes it verbatim AND adds a note directing agents to prefer Phase 15's `POST /api/tasks/:id/checkpoints` (allowlist entry #7) instead. This preserves doc/source truth-sync without the author modifying preamble.ts.
- **PUT /api/tasks/ mentions appear three times — all in drift-guard callouts.** The plan's verification regex was `! grep -q "submit → done"` (passes). The doc does contain the string `PUT /api/tasks/` — but only in three places, ALL of which are negative drift guards telling authors not to use it. This matches the plan's drift-guard section which lists `PUT /api/tasks/:id` as an explicit "do not" example.
