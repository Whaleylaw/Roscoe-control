<objective>
Build the copy-paste verification harness that keeps docs/runtime/ in sync with source. This harness is the ONLY anti-drift mechanism in the phase — it runs as part of `pnpm test:all` (or standalone via `pnpm docs:verify-runtime`) and fails when a claim in a doc no longer matches the source-of-truth file it cites.

Purpose: Without this harness, the new docs will drift the moment v1.3 lands (same way the Phase-14-era design spec drifted once Phase 17-01 flipped submit→review). The researcher made this explicit: "The critical risk is documentation drift — the design spec says things the code no longer does... Every documented claim must be validated against the current source files."

Output:
- `scripts/verify-runtime-docs.mjs` — Node ESM, uses only Node 22+ built-ins (`fs`, `path`, `child_process`) + existing repo deps. No new npm packages.
- `scripts/__tests__/verify-runtime-docs.test.mjs` — Vitest unit tests exercising the harness logic on fixture inputs.
- `package.json` — add `"docs:verify-runtime": "node scripts/verify-runtime-docs.mjs"` to scripts block.

Scope guardrail: This is the ONLY plan in Phase 18.1 that writes NEW src-side files. The new files live under `scripts/` (read-only harness, no runtime code). No tests under `src/lib/__tests__/` modified. No migration. No schema. No API route added.
</objective>

  <action>
Create `scripts/verify-runtime-docs.mjs` as a Node 22+ ESM script with NO new npm dependencies. Use only built-ins (`node:fs`, `node:path`, `node:child_process`, `node:url`) plus existing repo deps if convenient (e.g., `zod` for schema validation in --check recipes). All imports use ESM syntax.

## CLI shape
```
node scripts/verify-runtime-docs.mjs [--check <name>] [--verbose]

Checks (default: all):
  recipes, runner-env, agent-allowlist, agent-env, admin-config,
  ui-testids, tutorial-syntax, links, code-fences, drift-guard
```

## Output shape
- On success: one-line `ok` summary and exit 0.
- On failure: structured per-check report to stderr with file paths + line numbers + actionable fix hints, then exit 1.

## Architecture
- Top of file: a MANIFEST constant mapping each check name to an async function returning `{ check, ok, failures: string[] }`.
- A main() function iterating the requested checks (parsed from argv).
- Helper functions:
  - `readDoc(relPath)` → string (read docs/runtime/<name>.md)
  - `readSource(relPath)` → string
  - `extractFencedBlocks(markdown, langHint)` → Array<{lang, content, startLine}>
  - `extractMarkdownTables(markdown, heading)` → Array<Array<string>> (rows)
  - `resolveRelativeLink(docPath, linkHref)` → {absolutePath, anchor?}
  - `assert(cond, message)` helper that pushes to failures array rather than throwing — lets a check report ALL failures in one run

## Check implementations

### --check recipes
- Read docs/runtime/recipes.md
- Extract every fenced ```yaml block
- For each block, try to parse as recipe.yaml against `recipeYamlSchema` (import from `@/lib/recipe-schema` via relative path `../src/lib/recipe-schema.ts` — OR, to avoid tsx, dynamically import + use a JSON-ish yaml loader; simpler approach: spot-check field NAMES present in the block match known fields, and reject any block containing unknown top-level keys). Implement the simpler spot-check approach to avoid a TS runtime dependency.
- Fail with line numbers of any yaml block containing unknown top-level keys (not in recipeYamlSchema).

### --check runner-env
- Read docs/runtime/runner-daemon.md
- Extract the env-vars table (the table with columns Name/Type/Default/Source)
- For each Name row, grep scripts/mc-runner.mjs for `process.env.<NAME>`
- Fail with a list of env vars mentioned in docs but absent from source

### --check agent-allowlist
- Read docs/runtime/agent-contract.md
- Find the "Runner-token allowlist" table and extract the Method + Path pairs
- Read src/lib/runner-tokens.ts
- Extract entries from `export const RUNNER_TOKEN_ALLOWLIST = [ ... ]` (regex on `{ method: '<METHOD>', pathPattern: /<pattern>/ }`)
- Assert exact count parity (doc rows == source entries). Fail with a diff.

### --check agent-env
- Read docs/runtime/agent-contract.md
- Extract the MC_* env-vars table (columns Name / Type / Source-of-truth / Notes)
- Grep docker/hello-world-agent/README.md + src/lib/runner-claim.ts + scripts/mc-runner.mjs for each MC_* name
- Fail with env vars mentioned in docs but absent from all three source files

### --check admin-config
- Read docs/runtime/admin-config.md
- Extract all `runtime.*` keys mentioned (regex `runtime\.[a-z_]+`)
- Read src/app/api/settings/route.ts
- Extract all setting keys from the settingDefinitions object (regex `'runtime\\.[a-z_]+':\\s*{`)
- Fail with any key in docs but not in source (or vice versa for completeness)

### --check ui-testids
- Read docs/runtime/task-board-surfaces.md
- Extract every `data-testid="<value>"` pattern cited
- For each, read the component file referenced IN THE SAME SECTION (e.g., the RecipeBadge section cites src/components/panels/task-card/recipe-badge.tsx)
- Grep that component for `data-testid="<value>"`
- Fail with any data-testid cited in docs but absent from the component

### --check tutorial-syntax
- Read docs/runtime/getting-started.md
- Extract every fenced ```bash block
- For each block, do a lightweight syntax check: reject blocks where:
  - Commands end mid-heredoc (unterminated `<<`)
  - Obvious typos (e.g., `pnpm mc settngs`)
  - `curl` commands without a URL
- Do NOT execute the blocks (no side effects) — this is a sanity check only.

### --check links
- Read every docs/runtime/**.md file
- Extract every Markdown link target `](...)`
- For each non-http target, resolve relative to the containing doc's directory
- Assert the target file exists. For anchors (`#section`), best-effort check that a matching heading exists in the target file (normalize heading-to-slug via lowercase + dashes).
- Fail with a list of broken links grouped by source doc.

### --check code-fences
- Read every docs/runtime/**.md file
- Extract fenced blocks by language:
  - json → JSON.parse; fail with file:line on parse error
  - yaml → best-effort key:value structure check (every non-blank, non-comment line matches `^\s*[a-zA-Z0-9_.-]+:` OR is a continuation); fail with file:line on malformed
  - typescript → skip (we just confirmed the language hint; no syntax check)
  - bash → skip (covered by tutorial-syntax)
- Fail with per-block violations.

### --check drift-guard
- Read every docs/runtime/**.md file
- Assert NONE of these patterns appear:
  - `indexed_error` (legacy column name — renamed to `error_message` in Phase 18-04)
  - `submit → done`, `submit to done`, `submit.*flips.*done` (legacy lifecycle — Phase 17-01 locked submit→review)
  - `PUT /api/tasks/` inside agent-contract.md (legacy endpoint pattern — the runner-token allowlist rejects it)
  - `http://localhost` OR `http://127.0.0.1` inside fenced bash blocks of agent-contract.md (must use $MC_API_URL — pitfall 5)
- Fail with per-file line numbers of each offending hit.

## main() orchestration
- Parse argv; determine selected checks (default = all)
- Run each check; collect `{ check, ok, failures }`
- If all ok: print `[verify-runtime-docs] ok — all N checks passed` and exit 0
- If any failed: print a per-check report with file paths + line numbers + failure messages; print a summary counts line; exit 1

## Writing rules
- ESM (`import ... from ...`, no `require`). File begins with `#!/usr/bin/env node`
- No new npm dependencies; built-ins only
- Zero side effects (no writes, no network calls beyond reading files) — the harness is read-only
- Exit codes: 0 = ok, 1 = check failure, 2 = harness bug (caught exception)
- CLI help: `--help` prints usage and exits 0
- Conventional Commits; no AI attribution
