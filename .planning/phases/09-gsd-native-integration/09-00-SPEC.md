# 09-00 PLAN — Native GSD Integration in Mission Control (Option B)

Goal
Build first-class GSD lifecycle support directly into Mission Control so every project can be created, tracked, and executed through Discuss → Plan → Execute → Verify phases in one system.

Scope
- Native data model support for GSD at project + task level
- API support for lifecycle transitions and phase bootstrapping
- UI support in Project Manager / Project detail views
- Guardrails: approval gates before external-impact execution
- Test coverage for CRUD + transitions + idempotency

Non-Goals (for this phase)
- Full external automation to Paperclip/OpenClaw dispatch from MC UI clicks
- Rewriting existing GSD CLI internals in this pass
- New billing/tenant features unrelated to lifecycle

----------------------------------------------------------------
## Architecture (target)

A) Project-level GSD fields (native)
Add columns on `projects`:
- gsd_enabled INTEGER NOT NULL DEFAULT 0
- gsd_track TEXT NULL                 -- ops | product | marketing | legal | firmvault | custom
- gsd_phase TEXT NOT NULL DEFAULT 'discuss'  -- discuss | plan | execute | verify | done
- gsd_gate_mode TEXT NOT NULL DEFAULT 'manual_approval' -- manual_approval | auto_internal
- gsd_project_id TEXT NULL            -- canonical external ID / slug
- gsd_updated_at INTEGER NULL

B) Task-level GSD fields (native)
Add columns on `tasks`:
- gsd_phase TEXT NULL
- gate_required INTEGER NOT NULL DEFAULT 0
- gate_status TEXT NOT NULL DEFAULT 'not_required' -- not_required | pending | approved | rejected
- gate_approved_by TEXT NULL
- gate_approved_at INTEGER NULL
- depends_on_task_ids TEXT NULL       -- JSON array of task IDs

C) New API endpoints
1) POST /api/projects/:id/gsd/bootstrap
- Creates default phase task pack for the project
- Idempotent by phase (safe to rerun)

2) POST /api/projects/:id/gsd/transition
- Request body: { to_phase, reason }
- Enforces transition rules
- Writes activity/audit entries

3) PATCH /api/tasks/:id/gate
- Request body: { gate_status: approved|rejected, note }
- Records approver + timestamp

D) Transition rules
- discuss -> plan: requires at least one discuss-summary task done
- plan -> execute: requires at least one approved plan package task
- execute -> verify: requires all execute tasks done OR explicitly waived
- verify -> done: requires verification checklist task done

E) Approval guardrail
- Any task with gate_required=1 cannot move to in_progress/done unless gate_status=approved

## Default phase task templates (bootstrap)

Discuss phase (wave 1)
- DISCUSS-01: Clarify goal/scope/success criteria
- DISCUSS-02: Identify constraints + risks

Plan phase (wave 2)
- PLAN-01: Draft implementation plan
- PLAN-02: Approval package

Execute phase (wave 3)
- EXEC-01: Core implementation task(s)
- EXEC-02: Integration task(s)
(gate_required=1 for external-impact items)

Verify phase (wave 4)
- VERIFY-01: Verify acceptance criteria
- VERIFY-02: Ship/readout

----------------------------------------------------------------
## Acceptance Criteria

1) New project can be flagged gsd_enabled and assigned phase/track at create time.
2) Existing projects can be retrofitted with GSD metadata via PATCH.
3) Bootstrap endpoint creates phase tasks exactly once (idempotent).
4) Transition endpoint enforces lifecycle order and completion requirements.
5) Gate-required tasks cannot move to in_progress/done until approved.
6) UI exposes GSD controls without breaking existing project/task flows.
7) All new APIs appear in `/api/index` and OpenAPI if generated.
8) Tests pass for projects/tasks + new lifecycle behavior.

----------------------------------------------------------------
## Quick Manual Verification Checklist

1. Create project with gsd_enabled=true, gsd_phase=discuss.
2. Call bootstrap endpoint.
3. Confirm discuss tasks created.
4. Attempt discuss->execute transition (must fail).
5. Complete discuss criteria, transition to plan (must pass).
6. Create gate-required execute task.
7. Try setting task done without gate approval (must fail 403).
8. Approve gate, retry status update (must pass).
