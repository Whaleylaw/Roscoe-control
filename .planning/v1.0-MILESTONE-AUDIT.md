### FLOW-E partial gap (low severity)

`store.fetchProjects()` calls `GET /api/projects` without `?includeArchived=1`. Server filters `AND p.status = 'active'`. Result: after archiving a project via Settings, the next store refresh drops it from the `projects` array entirely rather than showing it with `status: archived`. Whether this is a gap depends on product intent:
- If archived projects should remain visible with a badge → **gap** (add `?includeArchived=1`)
- If archived projects should vanish from the active list → **correct behavior**

The PATCH itself works. No milestone requirement explicitly mandates archived-visibility UX, so flagged as tech debt, not a blocker.
